# Этап 1 — Host Shim: Windows

## Цель
Создать базовую платформенную обёртку (Host Shim) для операционной системы Windows. После завершения этого этапа CORE OS умеет открывать окно, получать события ввода с клавиатуры и мыши, и предоставляет абстрактный интерфейс для последующих платформ.

## Язык и стек
- **Язык:** Rust
- **Ключевые зависимости:** `winit` (окно + события), `windows-sys` или `windows` crate (Raw Input, DXGI), `wgpu` (инициализация позже, на этапе 7)
- **Целевая ОС:** Windows 10/11 (x64, ARM64)

## Зависимости
Это первый этап. Нет внешних зависимостей.

## Часть системы
**Level 0 — Host Shim** [См. layer-8 §4.1, layer-4 §3.1]

Host Shim — единственный слой, который напрямую взаимодействует с хост-ОС. Все остальные слои работают через его абстракции. На этом этапе реализуется только Windows-часть.

## Требования

### 1.1 Оконная подсистема
- Создание оконного контекста через `winit`.
- Поддержка оконного режима (windowed) и полноэкранного (borderless fullscreen).
- Управление размером, позицией, DPI scaling (per-monitor V2).
- Обработка событий: Resize, Move, Close, Focus, Minimize/Maximize.
- **Panic Gesture резервирован:** тройное касание угла или Ctrl+Shift+Esc должны генерировать специальное событие `PanicExit`, которое пробрасывается наружу. Реализация обработчика — в этапе 15 (Window Manager).

### 1.2 Ввод с клавиатуры
- Получение событий KeyDown / KeyUp / KeyRepeat.
- Поддержка всех клавиатурных раскладок (scancode → keycode → символ через OS layout).
- Модификаторы: Shift, Ctrl, Alt, Meta (Win).
- Комбинации: Ctrl+Alt+Del **не перехватывается** — это Secure Attention Sequence Windows, обрабатывается Winlogon [См. layer-4 §5.2].
- Сочетания Alt+Tab, Win+Tab в полноэкранном режиме: Alt+Tab переключает окна внутри CORE, Win/Super — открывает Command Bar [См. layer-1 §4.2, layer-4 §5.1].

### 1.3 Ввод с мыши
- События: Move, ButtonDown/Up, Scroll.
- Поддержка нескольких кнопок (до 5).
- Трекпад: multi-touch gestures (два пальца — скролл, три — назад/вперёд).

### 1.4 Абстрактный интерфейс Host Shim
- Определение trait/интерфейса `HostBackend`, который экспортирует:
  - `create_window(config) -> WindowId`
  - `poll_events() -> Vec<HostEvent>`
  - `set_cursor_style(style)`
  - `request_exit()`
- Этот интерфейс используется всеми последующими этапами. Windows-реализация — единственная «живая» на данный момент; остальные платформы реализуются в этапах 2–3.

### 1.5 Логирование и отладка
- Структурированное логирование через `tracing`.
- Логи пишутся в `%LOCALAPPDATA%\CoreOS\logs\`.
- Уровень логирования настраивается через переменную окружения `CORE_LOG`.

## Ключевые функции

| Функция | Описание | Тест |
|---------|----------|------|
| Окно 800×600 | Создание окна, видимость | Запуск → окно появляется |
| KeyDown 'A' | Нажатие клавиши, получение символа | Нажать A → лог содержит KeyDown(A) |
| Mouse click | Клик левой кнопкой | Клик → лог содержит ButtonDown(Left) + координаты |
| Resize | Изменение размера окна | Растянуть окно → лог содержит Resize(w, h) |
| Panic Gesture | Ctrl+Shift+Esc | Нажать → лог содержит PanicExit |
| DPI scaling | Перемещение между мониторами | Перетащить на 4K монитор → масштаб корректный |

## Интеграция с будущими этапами
- **Выход:** `HostEvent` stream → этап 7 (Display Server) получает события для рендеринга.
- **Выход:** `HostEvent` stream → этап 10 (Micro-Kernel IPC) получает ввод для обработки.
- **Вход:** от этапа 7 — запрос на создание surface для WebGPU.

## Критерии приёмки
- [ ] Компилируется на Windows x64 и ARM64.
- [ ] Окно открывается, закрывается, изменяет размер.
- [ ] Все события клавиатуры и мыши попадают в лог.
- [ ] Ctrl+Alt+Del не перехватывается (проверка через задачу безопасности).
- [ ] Panic Gesture генерирует событие.
- [ ] DPI scaling работает корректно на multi-monitor setup.
- [ ] Логи пишутся в файл.
