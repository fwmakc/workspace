# Layer 3 — System split | Разделение системы: Фронт и Бэк

Фронт (Shell) vs Бэк (Backoffice). API-граница, оптимистичные мутации, режимы работы (локально/P2P/оффлайн), мульти-Бэк (одновременное подключение к нескольким Space), корпоративный сценарий, администрирование (Core.Backoffice / Core.Hardcore).

## Принцип

CORE OS состоит из двух частей:

- **Фронт (Shell)** — всё, с чем взаимодействует пользователь. Рендеринг, ввод, выполнение приложений, кэш данных.
- **Бэк (Backoffice)** — источник истины. Данные, безопасность, синхронизация, индексация, контроль доступа.

Это разделение — **не опция, а фундамент**. Даже когда Фронт и Бэк работают на одном устройстве, это разные процессы, общающиеся через API. Архитектура с первого дня проектируется так, чтобы Бэк мог стоять на отдельном сервере, а Фронт — подключаться к нему по сети.

### Ключевое правило

**Бэк — источник истины, Фронт — кэш с правом рендеринга.**

- Код приложений: системные — зашиты в бинарник Бэка, сторонние — из App Registry на Бэке, веб-приложения (Island Mode) — загружаются из интернета
- Бэк контролирует доступ к данным на каждый запрос
- Фронт выполняет код локально (V8 Isolate), но все запросы к данным идут через API в Бэк
- При потере связи Фронт работает с кэшированными данными, мутации складываются в очередь
- При восстановлении связи — CRDT-синхронизация

## Фронт (Shell)

### Зона ответственности

Всё, что видит и трогает пользователь. Рендеринг, ввод, интерактивность.

### Компоненты

| Компонент | Что делает |
|-----------|-----------|
| Display Server | WebGPU-рендеринг всего UI. Композитит окна, применяет эффекты (blur, прозрачность) |
| Input Router | Клавиатура, мышь, тач, голос → маршрутизация в активное окно / Command Bar |
| Window Manager | Z-стек, фокус, перетаскивание, snap, тайловый/плавающий layout |
| Space Manager | Управление несколькими Space: размещение (плитки, мониторы, сплит), разделители, переходы между состояниями |
| Command Bar | Строка: Input Router, Suggestion Engine, 8 режимов. Своя в каждом Space, изолированная |
| V8 Isolates | Выполнение кода приложений. Каждое приложение — отдельный Isolate, привязанный к Space |
| Island Mode | Chromium sandbox для веб-контента (сайты как окна) |
| Voice Input | Whisper (локальный) → текст → Input Router |
| Profile Manager | **Фронтовая часть** split-компонента: freeze/thaw V8 Isolates, переключение layout при смене профиля, анонимный сессионный профиль |
| Data Cache | Локальный кэш данных из Бэка. Отдельный кэш на каждый Space |
| Optimistic Mutations | Мгновенное применение изменений в UI до подтверждения от Бэка |
| Storage Manager | Обнаружение внешних носителей, импорт/экспорт файлов. Связь с хост-ОС через Host Shim |

### Host Shim: связь с хост-ОС

CORE не пишет собственные драйверы для каждого устройства. Вместо этого Host Shim (Level 0, Rust) выступает адаптером между CORE и хост-ОС:

- **Окно и ввод:** winit / platform-native APIs (Win32, Cocoa, Wayland/X11, Android SurfaceFlinger)
- **GPU:** WebGPU через wgpu → использует драйверы хост-ОС (DirectX, Metal, Vulkan)
- **Файлы:** VFS Бэка → хост-ОС файловая система через стандартные API
- **Сеть:** WireGuard → сокеты хост-ОС
- **Аудио:** CPAL / platform audio APIs

Таким образом CORE «сидит поверх» хост-ОС, используя её инфраструктуру, но предоставляя свой UX.

### Чего Фронт НЕ делает

- Не хранит данные постоянно (только кэш)
- Не контролирует доступ (спрашивает Бэк)
- Не синхронизирует устройства (это делает Бэк)
- Не индексирует файлы (это делает Бэк)

### Принцип безопасности Фронта

V8 Isolate приложения не имеет прямого доступа к файловой системе и контактам. Сетевой доступ контролируется: уровни 1–2, 4–5 — только через API Бэка; уровень 3 — ограниченный доступ через whitelist доменов в манифесте (`core.json`). Единственный канал получения данных приложений уровней 1–2, 4–5 — API-запрос в Бэк. Бэк проверяет права и либо отдаёт данные, либо отказывает.

Даже взломанное приложение не может прочитать чужие данные — потому что Фронт их просто не имеет. Данные лежат в Бэке, доступ контролирует Бэк.

### AppRenderer — рендеринг приложений

Абстракция `AppRenderer` с двумя реализациями:

**LocalRenderer** — рендеринг на устройстве (по умолчанию). V8 Isolate + WebGPU. Приложение выполняется локально, рендерит через WebGPU pipeline.

**RemoteRenderer** — Бэк рендерит UI и стримит bitmap на Фронт. Для слабых устройств, которым нужно запустить тяжёлое приложение (видеоредактор, 3D). Как GeForce Now / Citrix.

Если устройство не тянет приложение — пользователь видит: "Это приложение требует более мощное устройство или Core Base с поддержкой удалённого рендеринга".

Архитектура закладывается с первого дня (единый интерфейс `AppRenderer`). Реализация RemoteRenderer — по мере необходимости, без привязки к версии.

## Бэк (Backoffice)

### Зона ответственности

Данные, безопасность, синхронизация, бизнес-логика. Бэк — хозяин всего, что имеет значение.

### Компоненты

| Компонент | Что делает |
|-----------|-----------|
| VFS + SQLite | Виртуальная файловая система. Хранение файлов и метаданных |
| CRDT-слой | Бесконфликтная репликация. Causal Trees, LWW, Hybrid Logical Clocks, Hash-based Ordering для true concurrent conflicts |
| P2P Mesh | WireGuard + libp2p + mDNS. Сетевой стек для связи устройств |
| Key Manager | Хранение ключей, генерация производных ключей. Пароли не покидают Бэк |
| Auth Proxy | Авторизация приложений. OAuth flow, выдача токенов. Приложения не видят пароли |
| Capability Security | Контроль доступа. Каждое приложение получает контекст с разрешениями |
| Search Engine | Полнотекстовый поиск (FTS5), OCR, контекстный поиск |
| Tag Engine | Теги, Smart Folders, граф связей между сущностями |
| App Registry | Хранение кода приложений, кэш, выдача по запросу Фронта. Модель приложений — 5 уровней интеграции, [Layer Apps](layer-6-apps.md) |
| Scheduler | Задачи по расписанию: напоминания, фоновые операции |
| Chat Engine | Мессенджер, почта, VoIP. Индексация переписки |
| Profile Manager | **Бэковая часть** split-компонента: хранение данных профилей, аутентификация, сессии, device key mapping. Исполнение (freeze/thaw, layout) — на Фронте |
| Sync Engine | Анти-энтропия: Merkle Search Trees, XOR-дельты, Zstd-сжатие |

### RBAC — роли и права доступа

С точки зрения RBAC, Space = Бэк. Подключение к Space = подключение к Бэку. Доступ определяется ролью на Бэке. Три встроенные роли: Owner (root), Member (чтение/запись), Guest (только чтение). Owner может создавать кастомные роли и группы. Назначение на уровне Space и проекта. Наследование: роль проекта ограничивает роль Space.

Подробно: роли, наследование, группы, управление — в [Layer Security, секция 3](layer-7-security.md).

### Аудит

Бэк логирует действия пользователей. Owner = root, полностью контролирует настройки аудита. 13 категорий (авторизация, роли, проекты, файлы, заметки, теги, мессенджер, поиск, приложения, браузер, профили, система, мульти-Бэк). Owner включает нужные категории, задаёт include/exclude фильтры, срок хранения, политику при переполнении.

Хранится в отдельной SQLite-таблице. Owner видит всё, Member/Guest — не видят.

Подробно: 13 категорий, настройки, JSON-конфиг, CLI, просмотр — в [Layer Security, секция 4](layer-7-security.md).

### Шифрование и безопасность

Owner выбирает уровень безопасности: базовый (шифрование диска), повышенный (zero-knowledge), максимальный (аппаратный ключ). Аутентификация устройств — без паролей, на ключевых парах (Ed25519). SSH — только по ключам.

Подробно: уровни безопасности, алгоритмы, Key Manager, Auth Proxy — в [Layer Security, секции 2](layer-7-security.md).

### User Directory — провайдеры пользователей

Бэк не привязан к конкретному источнику пользователей. Owner выбирает провайдер через абстракцию `UserDirectory`: Local (по умолчанию), LDAP (AD, FreeIPA), OAuth (Google, Microsoft, GitHub), Custom (плагин). Маппинг групп провайдера → роли CORE.

Подробно: провайдеры, настройка, техническая реализация — в [Layer Security, секция 5](layer-7-security.md).

### Конфигурация Бэка

Бэк — один бинарник. Разница только в наборе активных компонентов. Есть **ядро** (обязательные компоненты) и **опциональные компоненты**, которые включаются/отключаются под устройство.

**Ядро (обязательные, не отключаются):**

| Компонент | Причина |
|-----------|---------|
| SQLite | Хранение данных — без этого Бэк не работает |
| CRDT-журнал | Оффлайн-мутации — без этого не синхронизируется |
| Key Manager | Расшифровка кэша — без этого данные недоступны |

**Опциональные компоненты:**

| Компонент | По умолчанию | Заметка |
|-----------|-------------|---------|
| Chat Engine | Вкл | Приём сообщений, уведомления |
| Scheduler | Вкл | Напоминания, фоновые задачи |
| Search Engine | Зависит от устройства | Полнотекстовая индексация, расход батареи |
| P2P-сервер | Зависит от устройства | Обслуживание других устройств, расход батареи |
| App Registry | Зависит от устройства | Локальный кэш приложений |
| VoIP | Зависит от устройства | Звонки, тяжёлые кодеки |
| Tag Engine | Зависит от устройства | Индексация связей между сущностями |
| Sync Engine (сервер) | Зависит от устройства | Анти-энтропия для других устройств |
| Auth Proxy | Зависит от устройства | Авторизация приложений |
| Dev Tools (core-dev) | Выкл | CLI для разработки: install, run, logs, build, validate. [Layer Apps](layer-6-apps.md) |

**Профили настроек:**

| Профиль | Что включено | Для кого |
|---------|-------------|----------|
| Минимальный | Только ядро | Слабые устройства, максимум батареи |
| Сбалансированный | Ядро + Chat + Scheduler | Телефоны по умолчанию |
| Полный | Все компоненты | ПК, Core Base, серверы |

**Конфигурация:**
- При первой установке — мастер настройки, система рекомендует профиль по характеристикам устройства
- После установки — через Command Bar: `settings backoffice components`
- Удалённо — через Core.Hardcore (SSH + CLI)
- Компоненты можно включать/отключать динамически, без перезагрузки

**Принцип:** как Linux — есть ядро (обязательное), остальное — пакеты. Мощный планшет может включить P2P-сервер и быть узлом для других устройств. Дешёвый телефон — только ядро + чат.

## API-граница

Фронт и Бэк общаются через API. На одном устройстве — IPC (SharedArrayBuffer, наносекунды). Между устройствами — P2P (WireGuard, миллисекунды).

### Направление запросов

```
Фронт → Бэк:
  "Дай данные проекта X"
  "Сохрани заметку Y"
  "Найди файлы по запросу Z"
  "Отправь сообщение контакту W"
  "Дай код приложения Calculator"
  "Авторизуй доступ к GitHub"

Бэк → Фронт:
  Данные (содержимое файлов, заметок, задач)
  Код приложений
  Результаты поиска
  Push-уведомления (новые сообщения, напоминания)
  Подтверждение/отклонение мутаций
  CRDT-обновления от других устройств
```

### Фронт НЕ может

- Обратиться к диску напрямую (только через API Бэка)
- Обратиться к сети напрямую (только через API Бэка)
- Обойти capability-проверки (Бэк проверяет каждый запрос)

### Оптимистичные мутации

Фронт применяет изменения мгновенно в UI, отправляет в Бэк асинхронно. Но не для всех операций.

**Допустимые (оптимистично, без ожидания):**
- Заметки, задачи, теги — черновые данные, легко откатить
- Позиция окон, layout — визуальное состояние, не влияет на данные
- Черновики сообщений — ещё не отправлены
- Статус "прочитано" — некритично

**Недопустимые (ждём подтверждение Бэка):**
- Удаление файлов — необратимо
- Отправка сообщений/писем — нельзя "отозвать"
- Передача прав доступа — безопасность
- Любые безвозвратные действия над данными

**При отклонении Бэком:**
1. Подсветка красным в UI
2. Сообщение: "Бэк отклонил изменение: [причина]"
3. Автоматический откат к предыдущему состоянию
4. Изменение помечается как "отклонённое" в истории

**Индикатор в UI:**
- Пока Бэк не подтвердил — иконка синхронизации (часики) рядом с элементом
- Подтверждено — иконка исчезает
- Отклонено — красная иконка + уведомление

## Режимы работы

### Один ПК — всё локально

```
[Фронт] ←IPC→ [Бэк]
```

Фронт и Бэк на одной машине. IPC через SharedArrayBuffer — наносекунды. Пользователь не чувствует разделения. При этом процессы изолированы: падение Фронта не ломает Бэк, падение Бэка — не ломает Фронт (он переключается в режим кэша).

### ПК + телефон — Бэк на ПК

```
[Фронт телефона + Бэк (сбалансированный)] ←P2P/Wi-Fi→ [Бэк на ПК (полный)]
```

Телефон — Фронт с Бэком в сбалансированном профиле (ядро + Chat + Scheduler). Подключается к полному Бэку на ПК. Данные синхронизируются через CRDT. Задержка по Wi-Fi — 1-5ms, незаметна для пользователя.

### Core Base — всегда включённый сервер

```
[Фронт ПК] ←Wi-Fi→ [Бэк Core Base] ←Wi-Fi→ [Фронт телефона]
```

Core Base — коробочка в розетку. Полный Бэк, всегда доступен. Все устройства — Фронты, подключаются к нему. Работает даже когда ПК выключен.

### Оффлайн — нет связи с полным Бэком

```
[Фронт + Бэк (локальный профиль)] — автономно
```

Фронт работает с локальным Бэком (кэш + CRDT-журнал). Мутации складываются в CRDT-журнал. При подключении к полному Бэку — автоматический merge.

Файлы, не закэшированные ранее, недоступны. Пользователь видит: "Файл доступен только на домашнем ПК. Подключитесь для доступа."

## Сценарии

### 1. Один ПК, всё локально

Открытие заметки: Фронт → IPC → Бэк (проверка прав, выдача данных) → Фронт (рендеринг). Задержка наносекунды, неотличимо от монолита.

### 2. ПК + телефон по Wi-Fi

Телефон открывает проект: Фронт телефона → P2P → Бэк ПК (отдаёт layout + код приложений) → Фронт телефона (рендерит). Набор текста: каждый символ через P2P (1-5ms), быстрее чем задержка клавиатуры в браузере (10-30ms).

### 3. Телефон в метро, без сети

Фронт телефона работает с локальным Бэком (кэш + CRDT-журнал). Мутации в CRDT-журнал. При появлении сети — merge с полным Бэком.

### 4. Core Base — сервер в розетке

ПК выключен, телефон подключается к Core Base. Работает как с обычным Бэком. ПК включается — подключается к тому же Core Base, оба Фронта работают параллельно, CRDT синхронизирует.

### 5. Потеря связи во время работы

Бэк ПК недоступен (свет отрубили). Фронт телефона продолжает работу с локальным Бэком. ПК включается — автоматическая синхронизация.

### 6. Два пользователя, одно устройство

Профиль "Папа" и "Мама". Бэк загружает данные профиля, изоляция на уровне Бэка. Фронт не может прочитать данные другого профиля — Бэк не отдаст. Разделение Фронт/Бэк даёт изоляцию из коробки.

При смене профиля Clipboard очищается.

**Как создаётся второй профиль:**

Owner открывает настройки → «Добавить пользователя» → вводит имя → система генерирует recovery-фразу (24 слова, BIP-39) для нового профиля. Фраза показывается один раз, пользователь записывает. Из фразы derived profile key. Profile key шифруется на устройстве: `encrypt(profile_key, device_key + биометрия/PIN)` и сохраняется в системный keychain. При уровне «Базовый» биометрия не требуется — достаточно device key.

**Как пользователь входит в свой профиль:**

1. На экране входа — список профилей устройства («Папа», «Мама», «Анонимный»).
2. Выбор профиля → если уровень безопасности «Повышенный» или «Максимальный» — запрос биометрии/PIN.
3. Фронт загружает layout, проекты, приложения профиля.

### 7. Взломанное приложение
Приложение в V8 Isolate запрашивает чужой файл. Бэк проверяет capabilities → отказ. Isolate изолирован, прямого доступа к диску нет. Единственный канал — API Бэка, и он заперт.

## Корпоративный сценарий

### Общая схема

Компания. Часть сотрудников в офисе, часть удалённо. Арендованный сервер — корпоративный Бэк. У каждого сотрудника своё устройство с Фронтом и локальным Бэком.

```
                    [Корпоративный Бэк — арендованный сервер]
                     /           |            \
                    /            |             \
    [Фронт + Бэк]   [Фронт + Бэк]   [Фронт + Бэк]
     Офис, ПК          Удалёнка, ноут        Удалёнка, телефон
```

### Утро, все подключаются

Офисные — по локальной сети. Удалённые — через WireGuard-туннель через интернет. Каждый Фронт при запуске: "Дай мои проекты, layout последней сессии, непрочитанные". Бэк отдаёт. Рабочие столы как вчера.

### Совместная работа

Два человека в одном проекте, оба редактируют документ. CRDT гарантирует конвергенцию — изменения автоматически сходятся на всех устройствах. Для текста это merge без потерь, для состояний — Last Write Wins или Hash-based Ordering. Оба видят изменения друг друга в реальном времени.

### Удалёнщик с плохим интернетом

Фронт переключается на локальный Бэк. Работает с кэшем. Мутации в CRDT-журнал. Интернет появляется на 30 секунд — локальный Бэк синхронизирует накопившиеся мутации. Пропал — продолжает работать.

### Новый сотрудник

Устанавливает CORE → вводит корпоративный код приглашения (или QR) → Бэк создаёт профиль, назначает права → Фронт скачивает данные → через минуту готов к работе.

### Увольнение

Админ отзывает доступ → Бэк отзывает ключи устройства → Фронт получает "Доступ отозван" → локальный кэш зашифрован и недоступен без ключей Бэка → корпоративные данные не утекли.

### Корпоративные vs личные данные

Два профиля: "Работа" (подключён к корпоративному Бэку) и "Личное" (подключён к домашнему Бэку). Изоляция на уровне Бэка — Фронт физически не может смешать данные.

### Сервер упал

Все Фронты переключаются на локальные Бэки, работают с кэшем. Админ поднимает сервер из бэкапа. Все подключаются, CRDT-merge восстанавливает накопившиеся изменения. Ни байта не потеряно.

### Новое приложение для всей компании

Админ загружает приложение в App Registry на Бэке → назначает права ("всем" или "отделу X") → Фронты получают уведомление → пользователь нажимает → код загружается из Бэка, запускается в V8 Isolate.

Модель приложений: 5 уровней интеграции (от «набрал URL» до «полный натив на WebGPU»). App Registry, манифест `core.json`, app-scoped SQLite, `@core/*` API — в [Layer Apps](layer-6-apps.md).

### Иерархический P2P (Supernodes)

**Проблема:** Плоская mesh-сеть не масштабируется на 1000+ узлов. Каждый шлёт каждому — трафик растёт квадратично.

**Решение:** Иерархическая топология.

```
[Корпоративный Бэк — Supernode]
         |
    +----+----+
    |         |
[Отдел A]  [Отдел B]
 | | | |    | | | |
Узлы...    Узлы...
```

- **Supernodes:** Бэк-офисы на уровне отделов (или корпоративный сервер)
- **Внутри отдела:** локальная сеть (zero latency, mDNS)
- **Между отделами:** оптимизированные "мосты" с CRDT. Только Supernodes обмениваются глобальными дельтами.
- **Трафик:** не больше, чем у Slack, но данные остаются внутри периметра компании.

**Fallback:** Если Supernode недоступен — отдел переходит в автономный режим, работает с локальным кэшем. При восстановлении — CRDT-merge с центральным узлом.

## Администрирование

Два уровня управления Бэком.

### Core.Backoffice (GUI) — опциональное приложение

GUI-приложение внутри CORE. Работает через Фронт, подключённый к Бэку. Предустановлено в домашней конфигурации. Для повседневного управления: создать пользователя, назначить права, посмотреть статус.

**Корпоративный режим:** Core.Backoffice — не встроенный компонент, а приложение в App Registry. При установке Бэка с профилем "Corporate" — приложение не устанавливается. Вообще не существует на диске. Флаг в конфигурации Бэка: `allow_gui_admin: false` — даже при ручной загрузке Бэк отказывается запускать.

| Режим | Core.Backoffice | Core.Hardcore |
|-------|----------------|---------------|
| Домашний | Доступен из коробки | Доступен |
| Малая команда | Доступен | Доступен |
| Корпоративный | Заблокирован | Единственный способ |

### Core.Hardcore (TUI + CLI) — всегда доступен

TUI-приложение прямо на Бэке. Подключение по SSH. Без Фронта, без GUI, без WebGPU.

Для:
- Корпоративного сегмента (максимальная безопасность, нет администирования со стороны Фронта)
- Первичной настройки сервера (до подключения первого Фронта)
- Аварийного восстановления (сервер сломан, нужно починить)
- Автоматизации (SSH + CLI = скрипты, CI/CD)
- Аудита (SSH-сессия логируется)

Рядом с интерактивным TUI — CLI-режим для скриптовой автоматизации:

```
ssh admin@core-server -- core-cli user add --name "Иван" --role developer
ssh admin@core-server -- core-cli app deploy --file crm.pkg --role all
ssh admin@core-server -- core-cli backup --full --output /mnt/backup/
```

## Мульти-Бэк: одновременное подключение к нескольким Бэкам

### Проблема

Сотрудник подключён к корпоративному Бэку. Хочет слушать музыку из домашнего Бэка, открыть чат с семьёй — но всё это в его личном пространстве, а не корпоративном. Переключение профилей не помогает — пользователь хочет работать с обоими мирами одновременно.

### Решение

Архитектурно Фронт умеет подключаться к N Бэкам одновременно. Каждый Бэк — изолированное пространство (Space). Фронт рендерит окна из всех подключённых Space на одном экране.

```
[Фронт]
  ├── Space "Работа"  ← WireGuard → [Корпоративный Бэк]
  ├── Space "Дом"     ← P2P        → [Домашний Бэк на ПК]
  └── Space "Семья"   ← P2P        → [Семейный Бэк]
```

### Space — техническая реализация

Space — это подключение Фронта к конкретному Бэку. Пользовательские аспекты Space (размещение, виды, состояния) описаны в Layer 1. Здесь — техническая сторона.

Каждый Space во Фронте:
- Держит одно API-соединение с Бэком (IPC или P2P)
- Имеет отдельный кэш данных
- Содержит свои V8 Isolates (привязаны к Space, не могут обращаться к другим)
- Имеет свою Command Bar (изолированную, работает только с этим Бэком)
- Управляется Space Manager (размещение, разделители, переходы между состояниями)

### Изоляция между Space

Окно из одного Space не может обратиться к данным другого Space. V8 Isolate привязан к конкретному Space — все API-вызовы маршрутизируются только в его Бэк. Политики экспорта clipboard, копирования данных и ограничения — настраиваются Owner каждого Бэка. При конфликте политик побеждает более строгая.

Подробно: изоляция Space, clipboard-политики, копирование данных, векторы атак — в [Layer Security, секции 6–7](layer-7-security.md).

### Command Bar в мульти-Бэке

У каждого Space своя Command Bar, изолированная. Строка работает только с Бэком этого Space. Запросы не пересекаются между Space.

### Подключение к чужому Бэку — техническая реализация

Интерфейс описан в Layer 1 (код приглашения, QR, адрес). Технически:

1. **Владелец генерирует приглашение** — одноразовый токен (32 байта, `CORE-A7X9-K2M4`), привязанный к роли, с TTL (24 ч по умолчанию)
2. **Фронт нового устройства предъявляет токен** — через Command Bar / QR / прямой адрес
3. **Бэк верифицирует токен** — проверяет валидность, срок, роль. Если ок — начинает handshake
4. **ECDH Handshake:** Бэк и Фронт обмениваются публичными ключами → вычисляют общий session key → устанавливают WireGuard-туннель
5. **Бэк выдаёт credentials:** device key + refresh token, привязанные к устройству и роли
6. **Токен уничтожается** — одноразовый, повторное использование невозможно
7. **Дальнейшие подключения** — по device key, без повторного приглашения

**QR-код** содержит: адрес Бэка + токен приглашения. Одно сканирование — всё подключается.

**Прямой адрес** без приглашения: политика `allow_anonymous_connect` определяет, принимает ли Бэк (создаёт гостевую роль) или отклоняет.

### Политики Бэка

Каждый Бэк определяет свои политики. Не решение Фронта — решение владельца Бэка. Корпоративный Бэк может запретить подключение к другим Бэкам, запретить перетаскивание данных, или разрешить всё. Домашний — не ограничивает. При конфликте — побеждает более строгая.

Подробно: [Layer Security, секция 6.2](layer-7-security.md).

### Перетаскивание данных между Space

Зависит от политик обоих Бэков:

- Оба разрешили → можно
- Один запретил → нельзя
- Пользователь видит: "Корпоративная политика запрещает импорт внешних файлов"

### Уведомления в мульти-Бэке

Каждый Бэк отправляет push-уведомления подключённым Фронтам (новые сообщения, напоминания, системные события). Фронт собирает push от всех подключённых Бэков в единый поток.

- Каждое уведомление помечается идентификатором Space (иконка + цвет)
- Фронт применяет пользовательские приоритеты: прерывающие (звук, вибрация), тихие (значок), молчащие (без уведомлений)
- Пользовательские настройки приоритетов хранятся на Фронте, не на Бэке

### Оффлайн-Space

При потере связи с Бэком Space переключается в состояние "Оффлайн". Layout не ломается — окна остаются на месте.

- Серый оверлей поверх окон оффлайн-Space + сообщение: "Space 'Работа' недоступен. Работаете с кэшем."
- Можно работать с кэшированными данными: чтение, редактирование — мутации складываются в CRDT-очередь
- Файлы/проекты, которых нет в кэше — недоступны: "Файл доступен только при подключении к Space 'Работа'"
- Command Bar работает, но поиск только по локальному кэшу
- При восстановлении связи — оверлей снимается, CRDT-синхронизация автоматически

**Несколько Space одновременно:** онлайн-Space работает как обычно, оффлайн-Space — серый, но живой:

```
┌───────────────────┬───────────────────┐
│                   │ ░░░░░░░░░░░░░░░░ │
│  Space "Дом"      │ ░ Space "Работа" ░ │
│  (онлайн)         │ ░   Оффлайн      ░ │
│                   │ ░ Работаете с    ░ │
│  [🏠] [🔍]       │ ░ кэшем          ░ │
│                   │ ░░░░░░░░░░░░░░░░ │
└───────────────────┴───────────────────┘
```

### Копирование данных между Space

Бэк-контролируемая операция. Фронт — только триггер, не видит содержимое копируемых данных. Оба Бэка проверяют свои политики. Данные текут между Бэками напрямую (P2P/IPC), Фронт не участвует в передаче.

UI: drag-and-drop, Command Bar "скопировать файл X в Space 'Дом'", прогресс-бар. При отказе: "Корпоративная политика запрещает экспорт файлов".

Подробно: процесс, токены экспорта, ограничения — в [Layer Security, секция 6.4](layer-7-security.md).

### Мультибуфер обмена

Фронт хранит историю копирований — стек буфера обмена. Не один элемент, а несколько.

```
Буфер обмена (Space "Работа")
├── [1] "Текст из документа"          — 2 мин назад
├── [2] Ссылка: https://...           — 5 мин назад
├── [3] "Кусок кода из терминала"      — 12 мин назад
└── [4] Изображение (скриншот)         — 30 мин назад
```

**Управление:**
- Ctrl+C — копирует в стек как обычно (верхний элемент)
- Ctrl+V — вставляет верхний элемент
- Ctrl+Shift+V — показывает весь стек, стрелками выбираешь нужный
- Command Bar: "буфер обмена" → выпадает история копирований + настройки:

```
"буфер обмена"
├── 📋 История копирований
│   ├── [1] "Текст из документа"          — 2 мин назад
│   ├── [2] Ссылка: https://...           — 5 мин назад
│   └── [3] "Кусок кода"                   — 12 мин назад
└── ⚙️ Настройки
    ├── Размер стека: 20
    ├── Типы контента: Все
    └── Очистка при выходе: Нет
```

**Пользователь настраивает (комфорт):**

| Настройка | По умолчанию |
|-----------|-------------|
| Размер стека | 20 |
| Типы контента | Все (текст, изображения, файлы, ссылки) |
| Очистка при выходе из Space | Нет |

**Owner настраивает через политику Бэка (безопасность):** `clipboard_allow_export` (none/text/all), `clipboard_life_time`, `clipboard_persist`. Подробно: [Layer Security, секция 6.3](layer-7-security.md).

### Буфер обмена между Space

Внутри одного Space — буфер обмена работает без ограничений. Между Space — контролируется политикой `clipboard_allow_export` Owner каждого Бэка. По умолчанию: текст копируется свободно. Файлы, изображения — через механизм копирования между Space.

Подробно: политики, поведение при вставке — в [Layer Security, секция 6.3](layer-7-security.md).
