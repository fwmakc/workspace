## А. Противоречия

### А1. Profile Manager — Фронт или Бэк?

**Layer 2, строка 93:** Profile Manager перечислен как компонент **Бэка**.

Но из Layer 4 секция 8 видно, что Profile Manager делает:
- Freeze/Thaw V8 Isolates → это **Фронт**
- Переключение layout через Display Server → это **Фронт**
- Profile Store в SQLite → это **Бэк**

Profile Manager **разнесён** между Фронтом и Бэком, но Layer 2 заявляет его как чистый компонент Бэка.

**Решение:** В Layer 2 указать, что Profile Manager — split-компонент (данные на Бэке, исполнение на Фронте). В Layer 4 это уже отражено корректно.

---

### А2. «Код приложений загружается из Бэка» — не всегда

**Layer 2, строка 16:** `Код приложений загружается из Бэка`

Но:
- Island Mode (веб-приложения) грузят контент **из интернета**, не из Бэка
- Системные приложения (Core.Notes и т.д.) — откуда? В составе бинарника Фронта? Или Бэк их раздаёт? Нигде не уточнено.

**Решение:** Уточнить в Layer 2:
- Системные приложения — зашиты в бинарник Бэка, Фронт запрашивает код при первом запуске
- Сторонние приложения — из App Registry на Бэке (магазин)
- Веб-приложения (Island Mode) — загружаются из интернета, Бэк не участвует

---

### А3. «Не платит за облако» — неточно

**Layer 1, строка 403:** `Не платит за облако`

Но Layer 3:
- VDS — $20-50/мес (строка 707)
- S3-бэкап — платный
- Core Base — $100-200 (строка 739)

Layer 1 имеет в виду «не платит за iCloud/Google Drive как обязательную подписку», но формулировка абсолютная и противоречит Layer 3.

**Решение:** Смягчить в Layer 1: «Не платит за облако для своих данных — всё хранится на своих устройствах. Облачные сервисы (VDS, S3) — опционально, для продвинутых сценариев.»

---

### А4. «Закрыл — не жрёт память» — неточно

**Layer 1, строка 345:** `Закрыл — не жрёт память`

Но Layer 4 секция 4.3 (Lifecycle):
```
Активно → Приостановлено (Isolate заморожен, память удерживается) → Уничтожено
```

Закрытое приложение сначала переходит в Suspended (память удерживается для быстрого возврата). Память освобождается только при уничтожении.

**Решение:** Смягчить в Layer 1: «Закрыл — система сама решает, когда освободить память. Не жрёт ресурсы, когда не нужно.»

---

### А5. RemoteRenderer — «после выхода первой версии»

**Layer 2, строка 68:** `реализация RemoteRenderer — после выхода первой версии`

Нарушает дух правила «нет деления на MVP/post-MVP — всё идёт в первую версию». Пусть слово «MVP» не используется, смысл тот же: фича отложена.

**Решение:** Либо убрать упоминание сроков (архитектура закладывается, реализация — когда будет готова), либо признать, что некоторые фичи естественно выходят позже, и сформулировать это как «архитектурная готовность» без привязки к версии.

---

### А6. Space «Фоновый» — не описан технически

**Layer 1, строки 101:** Space «Фоновый» — «без графической области. Не занимает экран, но работает: качает файлы, принимает сообщения».

Layer 2 и Layer 4 **не описывают**, как работает фоновый Space:
- Как пользователь с ним взаимодействует?
- Через уведомления другого Space?
- Через Command Bar какого Space?
- Что происходит при voice-команде в контексте фонового Space?

**Решение:** Добавить в Layer 4 секцию 11 (Space Manager) описание фонового состояния: нет Display Server области, работает только Sync Engine + Notification Manager, взаимодействие через уведомления активного Space.

---

### А7. Как несколько пользователей на одном устройстве?

**Layer 2, строки 485-487:** «Профиль 'Папа' и 'Мама'. Бэк загружает данные профиля, изоляция на уровне Бэка.»

Но нигде не описан **процесс переключения**:
- Как пользователь B входит? Пароль? Ключ? Код?
- Если паролей нет (Layer 1: «без паролей»), как Бэк отличает Папу от Мамы на одном устройстве?
- Device key один на устройство — как он маппится на двух пользователей?

**Решение:** Добавить описание в Layer 2 или Layer 4. Предположительная модель: на одном устройстве — один device key, но несколько user sessions. Переключение — через список профилей (как вкладки). Бэк различает пользователей по session token, не по device key.

## Б. Дублирование между слоями

### Б1. Layer 2 ↔ Layer 4: массовое дублирование

Layer 4 (техническая декомпозиция) должен детализировать Layer 2 (Фронт/Бэк), но вместо этого **повторяет его почти дословно**:

| Тема | Layer 2 | Layer 4 | Проблема |
|------|---------|---------|----------|
| RBAC (роли, группы, scope) | Секция «RBAC» (~30 строк) | Секция 14 (~60 строк) | Одинаковое содержание |
| Аудит (13 категорий, конфиг) | Секция «Аудит» (~130 строк) | Секция 15 (~60 строк) | Одинаковое содержание |
| User Directory (LDAP/OAuth) | Секция «User Directory» (~30 строк) | Секция 16.1 (~40 строк) | Одинаковое содержание |
| Конфигурация Бэка (ядро + опциональные) | Секция «Конфигурация Бэка» (~40 строк) | Секция 16.2 (~30 строк) | Одинаковое содержание |
| Connection Protocol (токен, ECDH) | Внутри «Мульти-Бэк» (~20 строк) | Секция 17 (~50 строк) | Одинаковое содержание |
| Шифрование (3 уровня, алгоритмы) | Секция «Шифрование» (~30 строк) | Секция 10.3 (~40 строк) | Одинаковое содержание |
| Мультибуфер (стек, политики) | Секция «Мультибуфер» (~60 строк) | Секция 13 (~70 строк) | Одинаковое содержание |
| Копирование между Space | Секция «Копирование данных» (~40 строк) | Секция 13.3 (~20 строк) | Одинаковое содержание |

**Итого:** ~380 строк в Layer 2 и ~370 строк в Layer 4 — дублируют друг друга.

**Решение:** Layer 2 оставить **только концептуальное описание** (что, зачем, правила). Layer 4 оставить **только техническую реализацию** (как, потоки данных, уровни архитектуры, псевдокод). Конкретные шаги:
- Layer 2 «Аудит»: оставить 13 категорий + правила (кто видит, кто настраивает). Убрать JSON-конфиг, CLI-примеры, таблицу настроек — это Layer 4.
- Layer 2 «Шифрование»: оставить 3 уровня + принцип «без паролей». Убрать алгоритмы (AES-256-GCM, Ed25519, BLAKE3) — это Layer 4.
- Layer 2 «User Directory»: оставить 4 провайдера + принцип. Убрать bind_dn, client_id/secret — это Layer 4.
- Layer 2 «Connection Protocol»: оставить 7 шагов. Убрать технические детали ECDH, токен-генерацию — это Layer 4.
- Layer 2 «Мультибуфер»: оставить концепцию стека + политики. Убрать UI-примеры, настройки — это Layer 4.

### Б2. Layer 2 ↔ Layer 3: сценарии

**Layer 2 «Сценарии» (строки 463-492)** — 7 коротких сценариев Front/Back взаимодействия.
**Layer 2 «Корпоративный сценарий» (строки 495-539)** — детальный корпоративный кейс (45 строк).
**Layer 3** — 7 сценариев установки, многие покрывают ту же территорию.

Корпоративный сценарий Layer 2 и Layer 3 сценарии 6-7 — одна тема в двух слоях. Layer 2 фокусируется на Front/Back взаимодействии, Layer 3 — на установке/эксплуатации. Но по факту дублируют: утреннее подключение, совместная работа, увольнение.

**Решение:** Layer 2 «Сценарии» сократить до 5-строчных иллюстраций Front/Back взаимодействия (без сюжетов). Layer 2 «Корпоративный сценарий» убрать полностью — он полностью покрывается Layer 3 сценариями 6-7.

### Б3. Layer 2 ↔ Layer 3: безопасность

Layer 3 «Безопасность» (строки 72-168) и Layer 2 «Шифрование и безопасность» (строки 258-288) оба описывают 3 уровня безопасности и аутентификацию.

- Layer 2: архитектура (алгоритмы, где ключи)
- Layer 3: эксплуатация (векторы атак, защита)

Граница проведена, но с пересечением. «Кто что контролирует» (Layer 3 строки 151-167) — архитектурный вопрос, не сценарий установки.

**Решение:** Layer 2 — архитектурная модель (что зашифровано, где ключи, какие уровни). Layer 3 — операционная безопасность (векторы атак, защита, обновления). Убрать из Layer 3 таблицу «Кто что контролирует» — она ближе к Layer 2.

## Д. Пробелы — не описано нигде

### Д1. Множественные пользователи на одном устройстве

Как аутентифицируются, как переключаются, как device key маппится на разных людей. Layer 2 упоминает сценарий «два пользователя, одно устройство», но без технических деталей.

**Где добавить:** Layer 2 (концепция) + Layer 4 (реализация).

---

### Д2. Разрешения приложений (permissions UI)

Layer 4 показывает capability-based security с контекстом (fs, network, graphics, mind, contacts). Но нигде не описано:
- Где пользователь видит, какие права у приложения
- Как пользователь даёт/отзает права
- Что происходит при запросе нового разрешения

**Где добавить:** Layer 1 (пользовательский опыт) + Layer 4 (техническая реализация).

---

### Д3. Центр уведомлений

Layer 1 описывает push-уведомления и приоритеты Space, но не описывает:
- Где пользователь видит историю уведомлений
- Как очистить историю
- Можно ли вернуться к уведомлению (например, нажать на «новое сообщение от Ивана» и перейти в чат)

**Где добавить:** Layer 1 (UX) + Layer 4 (Notification Manager).

---

### Д4. Smart Folders

Layer 4 секция 5.2 описывает Smart Folders (папка = сохранённый поисковый запрос). Layer 1 не упоминает эту концепцию — пользователь не знает, что такие папки существуют.

**Где добавить:** Layer 1 (в секции «Теги вместо иерархии»).

---

### Д5. Обновление Фронта через Бэк

Layer 3 описывает обновление Фронта через App Store / автообновление при запуске. Но не описано:
- Может ли Core Base / сервер обновить Фронты подключённых устройств?
- Или Фронт всегда обновляется только через App Store / автообновление?

**Где добавить:** Layer 3 (в секции «Обновления»).

### Е1. Критичные угрозы

#### Е1.1. Уровень 3: свободная сеть + app-scoped data = эксфильтрация

**Слои:** layer-apps.md (уровень 3), layer-system-split.md

Бэкенд уровня 3 («Бэк на месте») получает sandboxed V8 Isolate с app-scoped SQLite, но при этом имеет **неограниченный HTTP/HTTPS доступ**. Приложение может выкачать все данные из app-scoped SQLite и отправить на любой внешний сервер.

Дополнительно: другие приложения (уровень 1-2, Island Mode, тоже свободная сеть) могут обратиться к `localhost:PORT` бэкенда уровня 3 и через его API выкачать данные.

**Решение:**
- Ограничить сетевой доступ уровня 3 через whitelist доменов в манифесте
- Firewall-правило: только frontend Island этого приложения может обращаться к `localhost:PORT`
- Network audit: логировать все исходящие HTTP-запросы уровня 3

---

#### Е1.2. Полное доверие хост-ОС (архитектурная)

**Слои:** все

CORE OS работает поверх хост-ОС и полностью ей доверяет. Host Shim — единственная точка контакта, но работает через API хост-ОС. Компрометация хост-ОС (rootkit, malware) = обход всех гарантий CORE OS. Хост-ОС может читать память CORE OS, перехватывать IPC, модифицировать файлы VFS. Нет integrity verification, нет Secure Boot, нет remote attestation.

**Решение:**
- Явно задокументировать риск: trusted computing base includes host OS
- Для максимального уровня безопасности: Integrity Monitoring Agent, периодически проверяющий хеш бинарника
- Remote attestation: Бэк может запросить у Фронта proof-of-integrity (TPM-based)
- Рекомендация: для чувствительных данных — dedicated device (Core Base)

### Е2. Высокие угрозы

#### Е2.1. V8 Bytecode: невозможность аудита безопасности

**Слой:** layer-apps.md (секция «Защита кода: V8 Bytecode»)

Приложения уровней 3-5 могут распространяться как bytecode — никто не может проверить что код делает. При этом уровень 3 имеет свободный доступ к сети. Сочетание «невозможность аудита + свободная сеть» = идеальный вектор эксфильтрации.

**Решение:**
- Runtime monitoring: все вызовы @core/* API и сетевые запросы перехватываются и логируются
- Политика `allow_bytecode_apps: false` для корпоративных Бэков (Owner может запретить)
- Магазин pkg.core.app принимает только исходный код; bytecode создаётся локально при установке
- Behavioural analysis: выявлять аномальные паттерны (массовые чтения + сетевые запросы)

---

#### Е2.2. Недостаточная sandbox-изоляция

**Слои:** layer-security.md (секция 7.3), layer-technical-decomposition.md

Указано «V8 Isolate + seccomp + отдельный процесс», но:
- Нет конкретики: какой seccomp-профиль? strict или filter? какие syscall разрешены?
- Нет упоминания namespaces (PID, network, mount, user)
- V8 имеет десятки CVE ежегодно — нет упоминания V8 Sandbox (memory corruption mitigation)
- Нет описания что происходит при обнаружении побега

**Решение:**
- Полный контейнерный sandbox: seccomp-bpf + user/mount/PID/net namespaces
- Внедрить V8 Sandbox (v8.dev/blog/sandbox)
- Каждый V8 Isolate — отдельный процесс с минимальным PID-namespace
- Runtime integrity checks: периодическая верификация памяти Isolate

---

#### Е2.3. Буфер обмена: свободный текст между Space

**Слои:** layer-system-split.md, layer-security.md

По умолчанию текст копируется между Space **свободно**. «Текст» может содержать IP-адреса, токены, пароли, финансовые данные. В корпоративном сценарии сотрудник может скопировать текст из корпоративного Space и вставить в личный. Нет опции «подтверждение при вставке из другого Space».

**Решение:**
- Для корпоративных Space: default `clipboard_allow_export: none` (не `text`)
- Добавить опцию «подтверждение при вставке между Space»
- Для корпоративных Space — обязательная очистка буфера при переключении Space

---

#### Е2.4. P2P/CRDT: нет rate limiting, нет undo

**Слои:** layer-technical-decomposition.md (секция 9), layer-security.md (секция 7.5)

CRDT гарантирует отсутствие конфликтов, но не защищает от вредоносных операций:
- Компрометированный узел может генерировать миллионы CRDT-операций в секунду → DoS
- Нет механизма отмены CRDT-операций (undo/rollback)
- Если device key скомпрометирован — нет отзыва уже выполненных операций
- Нет упоминания максимального размера CRDT-журнала или garbage collection

**Решение:**
- Rate limiting: max N операций в секунду от одного устройства
- CRDT garbage collection: операции старше M дней, уже merged — удаляются
- CRDT checkpoint: периодический снапшот состояния для отката
- CRDT quarantine: подозрительный поток от узла приостанавливается до подтверждения Owner

---

#### Е2.5. App Registry: нет модерации, нет kill switch

**Слои:** layer-apps.md, layer-system-split.md

Приложения попадают на устройство из нескольких источников без верификации содержания:
- pkg.core.app — нет описания процесса модерации
- Свой сервер разработчика — CORE OS проверяет только подпись и хеш
- Git-репозиторий — `core-dev install` может установить код откуда угодно
- Нет механизма отзыва вредоносных приложений (kill switch)
- Нет certificate pinning для pkg.core.app
- Нет анализа манифеста на подозрительные комбинации permissions

**Решение:**
- Mandatory code review + automated security scan для pkg.core.app
- Remote app kill: CORE Corp может пометить приложение как вредоносное, все Бэки автоматически удаляют
- Certificate pinning для pkg.core.app и update_url
- Анализ suspicious permission combinations: `fs` + `network` + `contacts` = высокий риск

---

#### Е2.6. Мультипользовательность: изоляция памяти между профилями

**Слои:** layer-system-split.md (сценарий 6), layer-technical-decomposition.md (секция 8)

Оба профиля работают в одном процессе Фронта. При freeze/thaw — V8 Isolates уничтожаются, но память может не быть обнулённой. Данные могут оставаться в SharedArrayBuffer (используется для IPC). Не описано: что происходит с clipboard при переключении профилей.

**Решение:**
- Secure memory wipe при freeze профиля: обнулять все V8 heap regions
- Clipboard автоматически очищается при переключении профиля
- Каждый профиль — отдельный SharedArrayBuffer region

---

#### Е2.7. Core Base / Raspberry Pi: нет шифрования диска

**Слои:** layer-installation-scenarios.md, layer-security.md

Для Core Base и Raspberry Pi: нет упоминания encrypted rootfs. Если вытащить SD-карту Raspberry Pi — всё на ней. Нет упоминания защиты при физическом подключении клавиатуры/монитора к Core Base.

**Решение:**
- Encrypted rootfs по умолчанию для Core Base и Raspberry Pi (LUKS / BitLocker)
- Boot-time passphrase через приложение CORE на телефоне (по QR)
- При подключении монитора/клавиатуры к Core Base — предупреждение Owner + аудит

### Е3. Средние угрозы

#### Е3.1. Фронт-кэш не зашифрован на украденном устройстве

**Слои:** layer-system-split.md, layer-security.md

При краже работающего устройства кэш Фронта доступен в открытом виде. При уровне «Базовый» ключ шифрования диска «на сервере» — на украденном устройстве кэш читается. При уровне «Повышенный» master key в RAM — возможна cold boot атака.

**Решение:** Шифровать кэш Фронта ключом от device key + screen lock. При простое > N минут — шифровать кэш. Remote wipe: Owner может удалённо затребовать уничтожение кэша.

---

#### Е3.2. core-dev: вектор установки произвольного кода

**Слой:** layer-apps.md (секция «core-dev»)

`core-dev install` принимает URL любого Git-репозитория. `--dev` даёт доступ к DevTools без ограничений. Не описано: кто может включить core-dev? Только Owner? Если включён на корпоративном Бэке — любой с SSH может установить произвольный код. `core-dev logs --api` показывает все API-вызовы — потенциальная утечка данных.

**Решение:** core-dev может включить только Owner. Для корпоративных Бэков — whitelist репозиториев. Логи core-dev показывают только API-вызовы своих приложений. Логи core-dev попадают в аудит.

---

#### Е3.3. Техподдержка: инсайдерская угроза

**Слои:** layer-installation-scenarios.md, layer-security.md (секция 1.3)

Сотрудник CORE Corp с доступом к HSM — единственная точка отказа. Уровень «Full» даёт полный доступ как Owner за 30 минут без dual approval. Нет упоминания: что если пользователь случайно подтвердит подключение (social engineering)?

**Решение:** Dual approval для уровня Full. Все сессии уровня Full — обязательная видеозапись. HSM key ceremony: минимум 2 из 3 сотрудников для подписи. Anomaly detection: если техподдержка обращается к одному Бэку > N раз — alert.

---

#### Е3.4. Обновления приложений: компрометация ключа разработчика

**Слой:** layer-apps.md (секция «Обновления приложений»)

Приватный ключ разработчика может быть скомпрометирован — нет механизма отзыва. Нет certificate pinning для update_url. DNS hijacking может подменить ответ. Нет требования HTTPS для update_url.

**Решение:** Registry of developer keys (CORE Corp может отозвать). Mandatory HTTPS. Certificate pinning при первой установке. `core-dev rollback` — только для Owner.

---

#### Е3.5. Голос: всегда включённый микрофон

**Слои:** layer-technical-decomposition.md (секция 7), layer-user-experience.md

Микрофон всегда активен. Нет упоминания аппаратного индикатора записи. Компрометация Host Shim = перехват аудио. Не описано: сохраняются ли аудио-фрагменты? Whisper model может быть подменена при обновлении.

**Решение:** Аппаратный индикатор записи (LED). Аудио-буфер обнуляется после распознавания. Пользователь может полностью отключить микрофон. Integrity check Whisper model (BLAKE3).

---

#### Е3.6. Owner может полностью отключить аудит

**Слой:** layer-security.md (секция 4)

Owner = root, может отключить аудит, выполнить вредоносные действия, включить обратно. Нет immutable audit trail — аудит хранится в SQLite, который Owner контролирует. Нет external audit log.

**Решение:** Immutable audit store (append-only WAL, Owner не может удалить записи). Опциональный external audit: отправка на syslog/SIEM. Для корпоративного профиля: аудит невозможно полностью отключить.

---

#### Е3.7. SMB/NFS: учётные данные через хост-ОС

**Слои:** layer-network.md (секция 2.1), layer-storage.md

Пароли SMB могут сохраняться в credential store хост-ОС. Host Shim вызывает `mount -t cifs` с паролем в командной строке — пароль виден в /proc.

**Решение:** Не сохранять SMB-пароли в credential store хост-ОС. Для Linux: credentials file, не command-line. Предупреждение: «Данные на сетевой папке не защищены CORE OS.»

---

#### Е3.8. Recovery-фраза: единственная точка отказа

**Слой:** layer-installation-scenarios.md (секция «Бэкап и восстановление»)

Recovery-фраза (24 слова BIP-39) — единственный способ восстановить данные. Украдена + stolen backup = полный доступ. Утеряна = невозвратимая потеря. Нет multi-signature recovery, нет Shamir's Secret Sharing.

**Решение:** Shamir's Secret Sharing: 24 слова делятся на 3 комплекта, нужно 2 для восстановления. Опция «social recovery» — доверенные контакты хранят части фразы. Для корпоративного: HSM-based recovery.

---

#### Е3.9. Error.message: целенаправленная эксфильтрация

**Слой:** layer-apps.md (секция «Отчёты об ошибках»)

Злоумышленное приложение может специально генерировать ошибки с пользовательскими данными в message. Rate limiting (10/час) ограничивает, но не исключает. Нет требования HTTPS для error_reporting_url.

**Решение:** Mandatory HTTPS для error_reporting_url. Уменьшить rate limit до 3/час. Пользователь видит preview полного отчёта перед отправкой.

---

#### Е3.10. CRDT-журнал на украденном устройстве

**Слои:** layer-system-split.md, layer-technical-decomposition.md

CRDT-журнал хранится локально. Не описано: зашифрован ли? При краже — история всех операций (содержимое заметок, файлов, сообщений) может быть прочитана.

**Решение:** CRDT-журнал шифруется device key + session key. При отзыве device key — журнал нечитаем. Pending мутации от отозванного устройства — Бэк отклоняет.

---

#### Е3.11. Анонимный профиль: данные в swap

**Слой:** layer-technical-decomposition.md (секция 8.3)

In-memory SQLite для анонимного профиля, но нет mlock()/VirtualLock(). Хост-ОС может сбросить страницу памяти на диск (swap/pagefile). После «удаления» данные могут оставаться в swap-файле.

**Решение:** mlock()/VirtualLock() для памяти анонимного профиля. Явное обнуление памяти перед освобождением. Рекомендация: для максимальной анонимности отключите swap в хост-ОС.

### Е4. Противоречия (безопасность)

#### Е4.1. «Нет доступа к сети» vs «Свободная сеть» для уровня 3

**Слои:** layer-system-split.md vs. layer-apps.md

layer-system-split.md: «V8 Isolate приложения не имеет прямого доступа к сети.»
layer-apps.md: уровень 3 бэкенд — «Сеть (HTTP/HTTPS) — Свободно».

Прямое противоречие. layer-system-split описывает общий принцип, но не учитывает уровень 3.

**Решение:** Уточнить в layer-system-split.md: «кроме уровня 3 backend, который имеет ограниченный сетевой доступ».

---

#### Е4.2. Уровень 3 менее безопасен чем уровень 4

**Слои:** layer-apps.md, layer-security.md

Уровень 3: свободная сеть, app-scoped SQLite. Уровень 4: сеть через @core/network (capability). Более «глубоко интегрированное» приложение имеет **больше свободы** и **меньше контроля**. Нумерация уровней (1-5) предполагает возрастание интеграции, но безопасность не возрастает монотонно.

**Решение:** Либо ограничить уровень 3 (whitelist доменов), либо явно документировать что уровень 3 — отдельная модель безопасности.

---

#### Е4.3. Auth Proxy: «приложение никогда не видит пароль» vs. уровень 3

**Слои:** layer-user-experience.md, layer-security.md vs. layer-apps.md

Auth Proxy описан как системный механизм, но уровень 3 с прямым сетевым доступом может обойти его, обратившись к внешнему сервису напрямую и перехватив OAuth-токен.

**Решение:** Явно указать ограничение Auth Proxy: не применим к уровню 3 (бэкенд имеет собственный сетевой стек).

---

#### Е4.4. Master key «никогда не покидает устройство» vs. «исчезает при перезагрузке»

**Слой:** layer-security.md (секции 1.1 vs. 2.1)

Если master key «исчезает при перезагрузке» (уровень «Повышенный»), как пользователь получает доступ к данным после перезагрузки? Нужен механизм re-derivation (через биометрию/PIN), но он не описан.

**Решение:** Уточнить: master key derived from biometrics/PIN stored in TPM/Secure Enclave. «Никогда не покидает устройство» верно. «Исчезает из RAM при перезагрузке» — уточнить что re-derived при входе.
