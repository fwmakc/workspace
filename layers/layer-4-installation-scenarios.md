# Layer 4 — Installation scenarios | Сценарии установки и эксплуатации

Как CORE OS устанавливается, обновляется и обслуживается в разных условиях: от одного телефона до корпоративного сервера.

---

## Принципы

### Обновления

**Фронт** обновляется автоматически через собственный канал:
- Мобильные устройства — через App Store / Google Play (автообновление)
- Десктоп — автообновление при запуске (проверка → скачивание в фоне → применение при перезапуске)

Бэк (включая Core Base и корпоративный сервер) **не обновляет Фронты** подключённых устройств. Обновление Фронта — решение устройства, не Бэка. Бэк может сообщить минимально совместимую версию, но не может форсировать обновление на чужом устройстве — это нарушило бы изоляцию.

Layout, проекты, кэш не ломаются при обновлении. Версия Фронта совместима с предыдущей версией Бэка (backward compatibility).

**Бэк** обновляется с подтверждением Owner. Перед обновлением — автоматический бэкап. При ошибке — откат к предыдущей версии. Owner может отложить обновление.

### Техподдержка

Сотрудник техподдержки **никогда не подключается к Фронту пользователя** — только к Бэку. Нет доступа к экрану пользователя, нет слежки.

**Ключевой принцип:** Бэк **глух** пока пользователь сам не инициирует техподдержку. Без инициативы пользователя — порт закрыт, запросы не принимаются, спам невозможен.

**Три способа вызвать техподдержку:**
1. Голос: "техподдержка" → Intent Parser → экран техподдержки
2. Command Bar: набрал "техподдержка" → экран техподдержки
3. Телефонный звонок: техподдержка просит продиктовать код → пользователь открывает CORE → видит код

**Процесс (простой — для любого пользователя):**
1. Пользователь говорит "техподдержка" или набирает в Command Bar
2. Бэк **открывает окно приёма** (TTL 10 мин) и генерирует код: `SUPPORT-9284-6173` (8 цифр)
3. Пользователь диктует код техподдержке по телефону
4. Техподдержка вводит код → relay-сервер CORE Corp пробивает NAT → WireGuard-туннель до Бэка
5. Бэк проверяет: окно открыто? код валидный? подпись CORE Corp верна?
6. На экране пользователя: "Подтверждённый специалист CORE Corp. Разрешить?" → одна кнопка
7. Пользователь подтверждает → сессия открыта (TTL 30 мин)
8. Пользователь видит "Специалист подключён" → может нажать "Отключить" в любой момент
9. Все действия логируются в аудит (категория "system")
10. Через 10 мин окно приёма закрывается → Бэк снова глух

**Relay-сервер CORE Corp:**
- Пробивает NAT — техподдержка может подключиться к любому Бэку за роутером
- Не хранит данные пользователей — только живые сессии (TTL 10 мин), потом удаляет
- Никакой регистрации устройств — Бэк сам подключается к relay при запросе техподдержки

**Подпись запросов:**
- Приватный ключ CORE Corp хранится в HSM (Hardware Security Module) — никогда не покидает железо
- Каждый запрос техподдержки подписан этим ключом (Ed25519)
- Бэк проверяет подпись публичным ключом CORE Corp (зашит в бинарник)
- Без валидной подписи — подключение невозможно, даже с правильным кодом

**Защита от атак:**
- Спам запросами — невозможен. Без валидного support-кода Бэк отвергает запрос, пользователь ничего не видит
- Перехват кода — код одноразовый, 100 млн комбинаций, TTL 10 мин
- Фишинг — пользователь сам инициирует. Без его действия окно приёма закрыто
- Подмена специалиста — подпись CORE Corp (HSM), подделать нельзя
- Взлом машины техподдержки — ключ в HSM, не извлекаем

**Уровни доступа:**

| Уровень | Что может | По умолчанию |
|---------|-----------|--------------|
| Read-only | Смотреть логи, конфиг, статус компонентов | Да |
| Diagnose | Read-only + запуск диагностических команд | По запросу |
| Full | Полный доступ (как Owner) | Только с явного согласия |

**Продвинутый режим (для гиков и админов):**
- SSH-доступ: `ssh support@back.address --support-code SUPPORT-9284-6173`
- Те же правила: код одноразовый, подпись CORE Corp, подтверждение Owner
- Whitelist IP для постоянной техподдержки (TTL на ключи, автоотзыв)
- Rate limiting: максимум 3 попытки с одного адреса, потом блокировка 30 мин

## Безопасность

CORE OS проектируется с подходом security by default: аутентификация без паролей (ключевые пары Ed25519), три уровня шифрования (базовый / повышенный zero-knowledge / максимальный с аппаратным ключом), RBAC с тремя встроенными ролями (Owner/Member/Guest), аудит 13 категорий, изоляция Space и приложений.

Полная модель безопасности — аутентификация, шифрование, RBAC, аудит, User Directory, изоляция, векторы атак и защита, обновления, техподдержка — описана в [Layer Security](layer-7-security.md).

**Минимальная поверхность атаки:** один открытый порт (WireGuard), SSH только по ключам, один пользователь ОС (`core`), процесс изолирован (seccomp, chroot), файловая система read-only где возможно.

**Обновления:** пакеты подписаны Ed25519 (ключ CORE Corp в HSM). Бэк проверяет подпись + хеш (BLAKE3) перед установкой. Автоматический бэкап перед обновлением. Откат при ошибке. Фронт обновляется автоматически, backward compatible.

## Бэкап и восстановление

### Принцип

Бэкап = зашифрованная копия данных Бэка. Даже если бэкап украли — без ключа Owner он бесполезен. Ключ шифрования бэкапа (backup key) генерируется из **recovery-фразы** — 24 слова, которые Owner записывает на бумаге при первом создании Бэка.

### Recovery-фраза

При создании Бэка CORE показывает:

```
Запишите эти 24 слова и храните в безопасном месте.
Это единственный способ восстановить данные при потере всех устройств.

word1 word2 word3 word4 word5 word6
word7 word8 word9 word10 word11 word12
word13 word14 word15 word16 word17 word18
word19 word20 word21 word22 word23 word24

⚠️ Никому не показывайте. Не фотографируйте. Не храните в цифровом виде.
```

Из recovery-фразы (BIP-39) генерируется backup key. Backup key шифрует бэкап. Без фразы — бэкап бесполезен.

**Как хранить recovery-фразу:**

| Способ | Для кого |
|--------|----------|
| Записать на бумаге, спрятать в сейф | Все |
| Файл на USB-флешке (зашифрованный) | Продвинутые |
| YubiKey / аппаратный ключ | Максимальная безопасность |

### Что бэкапим

- SQLite-базы (данные, настройки, аудит-лог)
- VFS-файлы (пользовательские файлы)
- CRDT-журнал (состояние синхронизации)
- Конфигурация Бэка (компоненты, политики, RBAC, User Directory)
- App Registry (кэш приложений)

### Чего НЕ бэкапим

- Device keys — генерируются заново при восстановлении
- Ключи сессий — временные, пересоздаются

### Куда бэкапим

**Правило 3-2-1:** 3 копии, на 2 разных носителях, 1 вне помещения.

Бэкап отправляется в **targets** — список целей. Каждая цель — адаптер с единым интерфейсом: `push(encrypted_data)`, `pull()`, `list()`, `delete()`.

**Встроенные адаптеры:**

| Адаптер | Куда | Как | Автоматически? |
|---------|------|-----|----------------|
| `usb` | Локальный диск / USB | Копирование файлов | Да, по расписанию |
| `core` | Другой Бэк CORE OS | P2P, WireGuard, CRDT-репликация | Да, постоянно |
| `s3` | S3-совместимое облако (Amazon S3, Яндекс Object Storage, Selectel, Backblaze B2) | Зашифрованная отправка по API | Да, по расписанию |
| `sftp` | Любой сервер по SSH/SFTP | Зашифрованная отправка | Да, по расписанию |

**Кастомные адаптеры (CustomTarget):**

Google Drive, Dropbox, OneDrive, WebDAV и любые другие хранилища — через плагины. Адаптер реализует интерфейс `BackupTarget` и регистрируется в App Registry. Owner или сообщество могут написать адаптер для любого хранилища.

```
BackupTarget (interface)
├── push(encrypted_data)  → отправить бэкап
├── pull()                → скачать последний бэкап
├── list()                → список доступных бэкапов
└── delete(id)            → удалить старый бэкап

Встроенные:
├── UsbTarget        — локальный диск / USB
├── CoreTarget       — другой Бэк CORE OS (P2P)
├── S3Target         — S3-совместимые хранилища
└── SftpTarget       — любой сервер по SSH/SFTP

Кастомные (через App Registry):
├── Google Drive adapter
├── Dropbox adapter
├── OneDrive adapter
├── WebDAV adapter
└── любой другой
```

### Настройки бэкапа

**Интерфейсы настройки:**

| Интерфейс | Когда |
|-----------|-------|
| Core.Backoffice (GUI) | Личный Бэк — Owner открывает приложение, настраивает бэкапы через GUI |
| Core.Hardcore (TUI/CLI через SSH) | Корпоративный Бэк — админ настраивает через CLI |
| Command Bar | Быстрая настройка: "настроить бэкап на USB" / "бэкап на S3: bucket, ключ" |

**Конфиг (JSON, хранится в Бэке):**

```json
{
  "backup": {
    "enabled": true,
    "schedule": "daily",
    "retention": 7,
    "targets": [
      { "type": "usb", "path": "/mnt/backup" },
      { "type": "core", "address": "core-base-2.local" },
      { "type": "s3", "bucket": "s3://my-backups", "region": "eu-west-1" },
      { "type": "sftp", "host": "backup.company.com", "path": "/backups/core" },
      { "type": "custom", "adapter": "google-drive", "config": { "folder_id": "..." } }
    ],
    "encryption": true,
    "warnings": {
      "no_backup_days": 3,
      "low_space_mb": 500
    }
  }
}
```

**Параметры:**

| Настройка | Что | По умолчанию |
|-----------|-----|--------------|
| targets | Куда бэкапить (USB / S3 / SFTP / другой Бэк) | Пусто (требуется настройка) |
| schedule | Каждый час / раз в день / раз в неделю | Раз в день |
| retention | Сколько копий хранить | 7 (неделя) |
| encryption | Всегда включено (backup key из recovery-фразы) | Всегда |
| warnings.no_backup_days | Предупредить если бэкап не делался N дней | 3 дня |
| warnings.low_space_mb | Предупредить если место заканчивается | 500 МБ |

### Предупреждения Owner

CORE предупреждает Owner, если:
- Бэкап не делался более 3 дней → "Бэкап не создавался 3 дня. Проверьте настройки."
- Recovery-фраза не записана → "Запишите recovery-фразу для восстановления данных."
- Место для бэкапа заканчивается → "Место на диске бэкапа заканчивается."

### Сценарии восстановления

#### Потерял телефон

1. Owner с другого устройства → отзывает device key потерянного телефона
2. Device key аннулирован → телефон больше не подключится
3. Кэш на телефоне зашифрован → данные не утекли
4. Новый телефон → установка CORE → подключение к Бэку → данные синхронизировались

#### Сменил ноут на новый

1. Старый ноут → отозвать device key (если доступен)
2. Или: с другого устройства → отозвать device key старого ноутбука
3. Новый ноут → установка CORE → подключение к Бэку → данные подтянулись

#### ПК (Бэк) сломался

1. Новый ПК / новое железо
2. Установка CORE Server
3. `core-cli restore --source /mnt/usb-backup/ --recovery-phrase`
4. Ввод recovery-фразы → backup key → расшифровка бэкапа → данные восстановлены
5. Устройства переподключаются к новому Бэку (код приглашения)

#### Core Base / сервер сгорел

1. Новый Core Base / новый сервер
2. Установка CORE Server
3. `core-cli restore --source s3://bucket/backups/ --recovery-phrase`
4. Или: с USB-диска / с другого сервера
5. Данные восстановлены, устройства переподключаются

#### Все устройства потеряны (пожар, кража)

1. Recovery-фраза записана на бумаге и хранится в другом месте (сейф, у родственника)
2. Покупаешь любое новое устройство
3. Установка CORE → "Восстановить из бэкапа" → ввод recovery-фразы
4. Бэк подключается к внешнему хранилищу (S3) → скачивает последний бэкап → расшифровывает
5. Данные восстановлены

## 1. Только телефон

**Устройство:** iPhone или Android.

### Установка

1. Скачивает CORE из App Store / Google Play
2. Открывает → "Введите своё имя"
3. Вводит → готово

**Что произошло:**
- Фронт + Бэк со сбалансированным профилем (ядро + Chat + Scheduler)
- Space "Home" с Owner
- Профиль пользователя, Home Project
- Бэк работает локально — SQLite, CRDT-журнал, ключи

### Повседневное использование

- Заметки, задачи, напоминания — всё на телефоне
- Чат с другими CORE-пользователями — P2P напрямую или через relay
- Фотографии, файлы — хранятся локально

### Ограничения

- Нет P2P-сервера — другие устройства не могут подключиться
- Нет полной индексации — поиск по кэшу
- Место ограничено памятью телефона

### Как расширить

Купил ПК → установил CORE на ПК → телефон подключился к Бэку на ПК → всё синхронизировалось.

### Обновление

Фронт + Бэк — вместе, через App Store / Google Play. Автоматически.

### Техподдержка

Бэк на телефоне недоступен по SSH извне (нет P2P-сервера). Но relay-сервер CORE Corp пробивает NAT:
- Пользователь говорит "техподдержка" или набирает в Command Bar
- Бэк генерирует код `SUPPORT-XXXXXXXX` → подключается к relay
- Пользователь диктует код техподдержке по телефону
- Техподдержка вводит код → relay → WireGuard-туннель → "Подтверждённый специалист CORE Corp. Разрешить?"
- Одна кнопка → сессия открыта
- Альтернатива: встроенный чат с поддержкой CORE в приложении (для несрочных вопросов)

## 2. Только ноут

**Устройство:** MacBook / Windows / Linux ноутбук.

### Установка

1. Скачивает CORE с сайта (`.dmg` / `.exe` / `.AppImage`)
2. Устанавливает
3. Открывает → "Введите своё имя"
4. Вводит → готово

**Что произошло:**
- Фронт + полный Бэк (все компоненты)
- P2P-сервер запущен — другие устройства могут подключаться
- Полная индексация, Search Engine, App Registry

### Повседневное использование

- Полноценная замена рабочего стола
- Ноутбук = личное облако для других устройств
- Закрыл крышку — Бэк спит. Открыл — всё на месте

### Как расширить

Купил телефон → установил CORE → телефон нашёл ноут по mDNS (одна Wi-Fi) → подключился → данные синхронизировались.

### Обновление

При запуске CORE проверяет обновление → скачивает в фоне → уведомление "Перезапустить для обновления" → подтверждение → обновление за 2-3 секунды.

### Техподдержка

Пользователь говорит "техподдержка" или набирает в Command Bar → код `SUPPORT-XXXXXXXX` → диктует техподдержке → relay пробивает NAT → подтверждение в UI. Если ноут в локальной сети — техподдержка может подключиться напрямую по SSH (продвинутый режим).

## 3. Домашний ПК и телефоны семьи

**Устройства:** стационарный ПК + телефоны мужа, жены, детей.

### Установка

**ПК (первый):**
1. Устанавливает CORE → вводит имя → Space "Home" создан
2. Owner = муж (тот, кто установил)
3. P2P-сервер, полный профиль Бэка

**Телефон мужа:**
1. Скачивает CORE → вводит имя
2. CORE находит ПК по mDNS → "Найден Бэк 'Home' на ПК. Подключиться?"
3. Подтверждает → ECDH handshake → подключение
4. Телефон = Фронт + сбалансированный Бэк, синхронизация с ПК

**Телефон жены:**
1. Скачивает CORE → вводит имя
2. Муж генерирует код приглашения: `CORE-K3M7-P2X9`
3. Жена вводит код → подключается к Бэку на ПК → роль Member (или Owner, если муж назначит)

**Телефон ребёнка:**
1. Скачивает CORE → вводит имя
2. Код приглашения → роль Member (или Guest на определённые проекты)
3. Ребёнок видит семейные проекты, но не рабочие

**Результат:**
```
[ПК, полный Бэк, Owner: муж]
  ├── [Телефон мужа, Фронт + сбалансированный, Owner]
  ├── [Телефон жены, Фронт + сбалансированный, Member]
  └── [Телефон ребёнка, Фронт + минимальный, Member/Guest]
```

### Повседневное использование

- Общий проект "Отпуск 2025" — все видят, все редактируют
- Фото с телефона → автоматически на ПК
- ПК выключен → все работают с кэшем на телефонах
- Муж на работе → жена и дети работают через Бэк на ПК

### Обновление

- ПК: при запуске — автообновление Бэка и Фронта
- Телефоны: через магазины приложений
- Owner может отложить обновление Бэка на ПК

### Техподдержка

- ПК — основной Бэк. Любой член семьи говорит "техподдержка" → код → диктует → relay → подтверждение
- Проблемы с телефоном — техподдержка подключается к Бэку на ПК, смотрит логи всех устройств
- Муж (Owner) может сам зайти через Core.Hardcore и починить

## 4. Core Base + устройства

**Устройства:** Core Base (коробочка в розетку) + телефоны + ноутбуки + Smart TV + проектор через мини-ПК.

### Установка

**Core Base:**
1. Распаковал → воткнул в розетку → подключил к роутеру
2. Светодиод мигает → "Готов к настройке"
3. С телефона: CORE → "Найден Core Base. Настроить?" → мастер настройки
4. Или: `core.base.local` с любого устройства в сети
5. Имя, Owner-профиль, полный профиль Бэка → готово

**Телефоны / ноутбуки:**
- Устанавливают CORE → код приглашения от Owner → подключаются к Core Base

**Smart TV:**
- CORE TV (приложение для Smart TV) → подключается к Core Base
- Плеер (фильмы с Core Base), фотогалерея, видеозвонки на большом экране

**Проектор через мини-ПК:**
- Мини-ПК → CORE → подключается к Core Base
- Фронт рендерит на проектор — презентации, фильмы, совместный просмотр

**Результат:**
```
[Core Base, полный Бэк, всегда включён, Owner]
  ├── [Телефоны, Фронт + сбалансированный]
  ├── [Ноутбук, Фронт + полный Бэк]
  ├── [Smart TV, Фронт TV-профиль]
  └── [Мини-ПК + проектор, Фронт + сбалансированный]
```

### Повседневное использование

- Все устройства всегда синхронизированы — Core Base не выключается
- Фильм начат на телефоне → продолжил на Smart TV с той же секунды
- Презентация с ноутбука → на проектор через мини-ПК
- Фотографии с телефона → на Core Base → видны на Smart TV

### Обновление

- Core Base: Owner получает уведомление "Обновление Бэка v2.1.0" → подтверждает → Core Base обновляется автоматически. Откат при ошибке
- Устройства: Фронт обновляется через магазины / автообновление

### Техподдержка

- Core Base всегда доступен в локальной сети и через relay извне
- Пользователь с любого устройства говорит "техподдержка" → код → диктует → relay → подтверждение
- Техподдержка подключается к Core Base, диагностирует, настраивает
- Если Core Base полностью недоступен — сброс кнопкой на корпусе → переподключение

### Бэкап

- USB-диск подключается к Core Base → автоматический бэкап раз в день
- Или: S3 (если Core Base имеет доступ к интернету)
- Recovery-фраза записана на бумаге и хранится в безопасном месте
- При замене Core Base: новый Core Base → "Восстановить из бэкапа" → USB-диск → recovery-фраза → данные на месте

## 5. Путь гика — Raspberry Pi вместо Core Base

**Устройства:** те же, но вместо Core Base — Raspberry Pi 5 (или любой SBC).

### Установка

**Raspberry Pi:**
1. Скачивает CORE Server (headless-образ для ARM) — `.img` для SD-карты
2. Записывает образ на SD → вставляет → загружается
3. SSH: `ssh core@raspberrypi.local` → мастер настройки (TUI)
4. Или: `raspberrypi.local:8080` с браузера → веб-мастер
5. Полный профиль Бэка → готово

**Дальше — как в сценарии 4:** все устройства подключаются к Raspberry Pi.

### Отличия от Core Base

- Гик сам обслуживает железо (SD-карта может умереть — нужен бэкап)
- Может подключить USB-диски, RAID, DVB-тюнер, GPIO-датчики
- Может настроить WireGuard для доступа извне
- Open-source — может пропатчить, если хочет

**Результат:**
```
[Raspberry Pi 5, CORE Server (headless), полный Бэк, Owner]
  ├── [Телефоны, ноутбуки, Smart TV, проектор — как в сценарии 4]
  └── Дополнительно: GPIO, умный дом, кастомные плагины
```

### Обновление

Гик сам: `ssh admin@rpi.local` → `core-cli update --check` → `core-cli update --apply`. Или через cron — каждую ночь. Перед обновлением — `core-cli backup --full`.

### Техподдержка

Гик скорее всего сам себе техподдержка — SSH + Core.Hardcore напрямую. Если нужна помощь CORE Corp:
- Простой режим: "техподдержка" в Command Bar → код → relay → подтверждение
- Продвинутый: `core-cli support generate-code --level diagnose` → код → техподдержка подключается по SSH
- Гик может настроить WireGuard для удалённого доступа техподдержки из любой точки

### Бэкап

- Гик настраивает сам: USB-диск + cron → `core-cli backup --full --output /mnt/usb/`
- Или: S3 (если Raspberry Pi имеет доступ к интернету)
- Или: rsync на другой сервер
- Recovery-фраза записана на бумаге (гик понимает важность)
- При замене SD-карты / Raspberry Pi: `core-cli restore --source /mnt/usb/ --recovery-phrase`

## 6. Компания с сервером в кладовке

**Устройства:** сервер в кладовке/серверной + ноутбуки/ПК сотрудников.

### Установка

**Сервер:**
1. Устанавливает Ubuntu Server (или любой Linux)
2. `curl -fsSL https://core.app/install.sh | bash` или Docker: `docker run -d --name core-backoffice core/server:latest`
3. Мастер настройки через SSH (Core.Hardcore TUI):
   - Профиль: Corporate
   - User Directory: LDAP (AD) или Local
   - Security: Повышенный или Максимальный (zero-knowledge)
   - Аудит: включён, категории по потребности
4. Owner = админ

**Сотрудники (офис):**
1. Устанавливают CORE на рабочие ПК/ноутбуки
2. Корпоративный код приглашения: `CORE-CORP-X7K2` (роль Member)
3. Подключаются по локальной сети → данные подтягиваются
4. Или: LDAP → авторизация по корпоративному логину

**Сотрудники (удалённо):**
1. Устанавливают CORE
2. Подключаются через WireGuard-туннель (админ настроил)
3. Код приглашения или LDAP-логин
4. Работают как в офисе

**Результат:**
```
[Сервер в кладовке, CORE Server, полный Бэк, Owner: админ]
  ├── User Directory: LDAP (AD компании)
  ├── RBAC: Owner (админ), Member (сотрудники), Guest (стажёры)
  ├── Аудит: включён
  ├── Security: Повышенный (zero-knowledge)
  │
  ├── [Офисные ПК, Фронт + полный Бэк, по LAN]
  ├── [Удалённые ноутбуки, Фронт + сбалансированный, по WireGuard]
  └── [Удалённые телефоны, Фронт + сбалансированный, по WireGuard]
```

### Повседневное использование

- Утром все подключаются — Бэк отдаёт проекты, layout сессий
- Совместная работа — CRDT
- Сервер упал — все работают с кэшем, поднялся — синхронизация
- Новый сотрудник → код → через минуту работает
- Уволился → админ отзывает доступ → кэш зашифрован

### Обновление

- Админ: `core-cli update --check` → `core-cli update --apply` по SSH
- Автоматический бэкап перед обновлением
- Сотрудники — Фронт обновляется автоматически

### Техподдержка

- Внутренняя: админ компании = техподдержка. SSH к серверу по LAN (продвинутый режим)
- Внешняя (CORE Corp): админ говорит "техподдержка" → код → relay через WireGuard → подтверждение в UI
- Аудит-лог фиксирует все действия техподдержки
- Owner видит активную сессию, может отключить в любой момент

### Бэкап

- Внешний USB-диск на сервере → автоматический бэкап раз в день
- Второй сервер в другой кладовке / офисе → взаимный бэкап через P2P
- Recovery-фраза хранится у админа в сейфе + у директора (два человека, две копии)
- При выходе сервера из строя: новый сервер → восстановление с USB / с другого сервера → recovery-фраза → данные на месте

## 7. Компания с VDS в дата-центре

**Устройства:** VDS в облаке (Hetzner, DigitalOcean, Яндекс.Облако) + устройства сотрудников.

### Установка

**VDS:**
1. Арендует VDS (минимум: 2 vCPU, 4 GB RAM, 50 GB SSD)
2. Устанавливает CORE Server — тот же процесс, что в сценарии 6
3. Публичный IP → доступен из любой точки
4. WireGuard для безопасного подключения Фронтов
5. Security: **Максимальный** — zero-knowledge + аппаратный ключ у админа
6. Бэкап: `core-cli backup --full --output /mnt/backup/` → S3 / другой сервер

**Сотрудники:**
- Те же шаги, что в сценарии 6
- Подключаются через интернет (WireGuard)
- Бэк доступен 24/7

**Результат:**
```
[VDS в дата-центре, CORE Server, полный Бэк, Owner: админ]
  ├── Security: Максимальный (zero-knowledge + YubiKey)
  ├── Публичный IP + WireGuard
  ├── Бэкап: S3 / удалённый сервер
  │
  ├── [Офисные ПК, по WireGuard]
  ├── [Удалённые сотрудники, по WireGuard]
  ├── [Телефоны, по WireGuard]
  └── [Командировки, кафе, метро — по WireGuard]
```

### Отличия от сервера в кладовке

- Доступен из любой точки — не нужен NAT/проброс портов
- Надёжнее — дата-центр = бесперебойники, резервные каналы
- Дороже — аренда VDS ($20-50/мес)
- Безопасность критичнее — данные в чужом дата-центре, обязателен zero-knowledge

### Обновление

VDS всегда доступен. Админ: `core-cli update --apply` по SSH. Или автообновление с подтверждением Owner. Бэкап → на S3.

### Техподдержка

- VDS доступен через интернет — самый простой сценарий
- Админ говорит "техподдержка" → код → техподдержка подключается через relay или напрямую (публичный IP)
- Подпись CORE Corp проверяется Бэком, подтверждение Owner в UI
- При постоянной поддержке: админ добавляет публичный ключ SSH сотрудника в whitelist (TTL на ключи, автоотзыв)
- Owner контролирует все сессии через аудит-лог

### Бэкап

- S3 (Amazon / Яндекс / Selectel) → автоматический зашифрованный бэкап раз в час
- + локальный бэкап на другой сервер → раз в день
- Recovery-фраза хранится у админа в сейфе + у директора (два человека, две копии)
- При выходе VDS из строя: новый VDS → `core-cli restore --source s3://company-backups/ --recovery-phrase` → данные восстановлены
- Zero-knowledge обязателен — данные в чужом дата-центре, бэкап тоже зашифрован
