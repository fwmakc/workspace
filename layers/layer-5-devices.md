# Layer 5 — Devices | Устройства и носители

Как Workspace работает с внешними устройствами и ресурсами: физическими носителями (USB, диски, оптика), сетевыми подключениями (P2P Mesh, SMB/NFS, принтеры), импортом/экспортом.

---

## Принцип

Workspace **никогда не работает с дисками напрямую**. Всегда поверх хост-ОС (Windows, Linux, macOS, Android). Хост-ОС управляет драйверами, монтирует файловые системы, обеспечивает доступ к блочным устройствам.

**Host Shim (Rust, Level 0)** — единственный слой Workspace, контактирующий с файловой системой хоста. Через стандартные API хост-ОС:
- Windows: `CreateFile`, `ReadFile`, `WriteFile`, `FindFirstFile`
- Linux: `open`, `read`, `write`, `readdir`
- macOS: те же POSIX API

Приложения, Space, Command Bar — **никогда** не обращаются к файлам хост-ОС напрямую. Только через API Бэка. Бэк — через Host Shim.

### Сетевое взаимодействие

Два уровня сетевого взаимодействия:

1. **WORKSPACE-to-WORKSPACE** — Workspace реализует сама. P2P Mesh (WireGuard + libp2p + mDNS). Полный контроль: обнаружение, подключение, синхронизация, стриминг, шифрование.

2. **WORKSPACE-to-World** — через хост-ОС. Workspace **не реализует** сетевые протоколы (SMB, NFS, DLNA, IPP, AirPlay, Chromecast). Хост-ОС делает грязную работу — монтирует файловые системы, управляет принтерами, подключается к Wi-Fi. Workspace видит результат через Host Shim.

**Host Shim — единственная точка контакта** с сетевыми ресурсами хост-ОС. Приложения, Space, Command Bar — никогда не обращаются к сети хост-ОС напрямую.

## Как работают файлы

### От пользователя к железу

Ты добавляешь файл в WORKSPACE. Что происходит под капотом:

1. **Passport + Body** — файл разделяется на две части:
   - **Passport** (SQLite): имя, проект, теги, права, CID (BLAKE3-хеш содержимого). Весит байты.
   - **Body** (blobs/): реальные байты файла, лежат по адресу CID. См. [секция 6.3](#63-где-хранятся-файлы-vfs).

2. **Синхронизация на другие устройства** — через CRDT-журнал (см. [Layer 3](layer-3-system-split.md)):
   - Passport (метаданные) — синхронизируется мгновенно. На телефоне ты сразу видишь файл в проекте.
   - Body (реальные байты) — **не копируется автоматически**. На телефоне создаётся **ghost-запись**.

3. **Ghost-файл** — запись в passport есть, CID известен, но blob отсутствует локально:
   - Ты видишь файл в проекте, можешь искать его по тегам, видеть превью (если есть), но он занимает **0 байт** на устройстве.
   - `available_local: false` — файл помечен облачком в интерфейсе.

4. **Открытие файла** — Lazy Load Engine запрашивает body:
   - Ты нажал на фильм → система стримит его с устройства, где blob есть (домашний ПК, WORKSPACE Base), по мере просмотра через P2P Mesh (WireGuard).
   - Ты нажал на документ → blob скачивается целиком, кэшируется локально, в следующий раз открывается мгновенно.

5. **Prefetch и умный кэш** — система догадывается заранее:
   - Ты открыл файл 1 из папки → система подгружает файл 2, 3, 4 в фоне.
   - Ты слушаешь плейлист → треки качаются по Wi-Fi для офлайна.
   - Кэш живёт на устройстве до тех пор, пока не закончится место. Тогда удаляются редко используемые blob'и (ghost-записи остаются).

### Примеры

**Фильм на 50 ГБ:**
- Passport (имя, CID, проект) — мгновенно на всех устройствах.
- Body (50 ГБ) — остаётся на ПК. На телефоне — ghost-запись.
- Ты открыл на телефоне → стримится с ПК по мере просмотра. 50 ГБ **не скачиваются** целиком, если ты не досмотрел до конца.
- Если нажал «скачать для офлайна» — blob передаётся в фоне по Wi-Fi.

**1000 фотографий по 5 МБ (25 ГБ):**
- Passport'ы (1000 записей в SQLite) — мгновенно на всех устройствах.
- Body (25 ГБ) — на телефоне создаются 1000 ghost-записей. Реальные файлы **не качаются**.
- Ты открыл фото №1 → скачалось 5 МБ. Листаешь дальше → prefetch подгружает фото №2, №3, №4.
- Остальные 996 фотографий продолжают занимать 0 байт на телефоне.

### Ограничения

- **Только телефон** (без ПК/WORKSPACE Base): файлы хранятся локально, место ограничено памятью телефона. Ghost-файлы не работают — неоткуда стримить. См. [Layer 4 — сценарий 1](layer-4-installation-scenarios.md).
- **Оффлайн**: файлы без локального blob'а недоступны. Пользователь видит: «Файл доступен только на домашнем ПК. Подключитесь для доступа». См. [Layer 3 — оффлайн-режим](layer-3-system-split.md).

## 1. Как Host Shim обнаруживает носители

Host Shim подписан на события хост-ОС о подключении и отключении носителей:

**Linux:**
- `udev` — события добавления/удаления блочных устройств
- `/proc/mounts` — отслеживание точек монтирования
- `inotify` — изменения в директориях монтирования

**Windows:**
- `WM_DEVICECHANGE` — событие подключения/отключения устройства
- `DBT_DEVICEARRIVAL` / `DBT_DEVICEREMOVECOMPLETE`
- `GetLogicalDriveStrings` — список доступных дисков

**macOS:**
- `DiskArbitration` framework — события подключения/отключения дисков
- `DADiskAppear` / `DADiskDisappear`

**Событие → Host Shim → Бэк:**

```
Хост-ОС: "USB-накопитель подключён" (/mnt/usb или D:\)
  |
  v
Host Shim (Level 0):
  +-- Определяет тип: USB-флешка, внешний HDD, CD/DVD, сетевой диск
  +-- Определяет файловую систему: NTFS, FAT32, exFAT, ext4, CDFS, UDF
  +-- Определяет размер, свободное место, метку тома
  +-- Передаёт событие в Бэк: { type, path, fs_type, size, free, label }
  |
  v
Бэк (Storage Manager):
  +-- Классифицирует носитель
  +-- Генерирует уведомление для Фронта
  +-- Ждёт действие пользователя
```

## 2. Типы носителей

### 2.1 Внутренние (HDD/SSD)

Внутренние диски — основное хранилище. Хост-ОС монтирует их при загрузке. Workspace использует для:
- VFS (файлы пользователей) — директория в домашней папке пользователя хост-ОС
- SQLite (данные, метаданные, аудит)
- Кэш приложений (App Registry)
- CRDT-журнал

Пользователь **не думает о дисках**. VFS абстрагирует физическое расположение. Файлы, проекты, теги — всё в VFS, где конкретно на диске — решает Бэк.

Пользователь видит только:
- «Место: 847 ГБ свободно из 1 ТБ» — в настройках
- «Место заканчивается» — предупреждение если < 10%

**WORKSPACE Base / Raspberry Pi:** хост-ОС (минимальный Linux, например Arch) монтирует внутренний диск. Workspace использует его так же — через Host Shim. Никакой разницы в логике.

### 2.2 Внешние (USB-флешки, внешние HDD/SSD)

Подключение → уведомление → действие пользователя.

**Уведомление:**

```
┌─────────────────────────────────────────┐
│  📀 Обнаружен USB-накопитель "FOTO-2025" │
│  32 ГБ, exFAT, 12 847 файлов            │
│                                          │
│  [Импортировать файлы]  [Бэкап на него]  │
│  [Игнорировать]                          │
└─────────────────────────────────────────┘
```

**Импортировать файлы** — файлы копируются в VFS Workspace:
- Каждый файл импортируется как новый объект VFS
- Автоматическая индексация (Search Engine)
- Автоматические теги по типу: фото → `#фото`, документы → `#документ`, музыка → `#музыка`
- Файлы привязываются к текущему проекту (или Home Project)
- Дедупликация: если файл уже есть в VFS (BLAKE3 хеш совпадает) — создаётся ссылка, копия не создаётся

**Бэкап на него** — запускается бэкап Бэка на этот носитель (см. [Layer: Установка, секция «Бэкап»](layer-4-installation-scenarios.md)).

**Игнорировать** — носитель не отображается в Workspace. Событие сохраняется в истории, повторное подключение — молча игнорируется. Пользователь может изменить через настройки: «Показывать игнорируемые носители».

**Извлечение:**

```
┌─────────────────────────────────────────┐
│  ⚠️ USB-накопитель "FOTO-2025"           │
│  Импорт ещё идёт: 340 из 12847 файлов   │
│                                          │
│  [Дождаться завершения]  [Отменить]      │
└─────────────────────────────────────────┘
```

Если идёт импорт — предупреждение. Если не идёт — носитель можно безопасно извлечь.

**Настройки автоматизации (опционально):**

Пользователь может задать правило для конкретного носителя (по метке тома):
- «Всегда импортировать фото с "FOTO-2025"»
- «Всегда делать бэкап на "BACKUP-1TB"»
- «Всегда игнорировать "WORK-USB"»

При повторном подключении — правило применяется автоматически, без уведомления.

### 2.3 Оптические (CD/DVD) — read-only

Минимальная поддержка. CD/DVD — устаревающий формат, но всё ещё используется в промышленности и архивах.

**Подключение → уведомление:**

```
┌─────────────────────────────────────────┐
│  💿 Обнаружен CD-диск                    │
│  700 МБ, CDFS, 23 файла                 │
│                                          │
│  [Импортировать файлы]  [Воспроизвести]  │
│  [Игнорировать]                          │
└─────────────────────────────────────────┘
```

- **Импортировать файлы** — как с флешки, копирование в VFS
- **Воспроизвести** — если аудио CD или видео DVD, открывается WORKSPACE Player
- Запись на CD/DVD **не поддерживается** — read-only

### 2.4 Сетевые хранилища (SMB/NFS)

Если хост-ОС подключил сетевой диск (SMB-шара, NFS-экспорт) — Workspace видит его как обычный путь через Host Shim.

Не отдельный тип носителя — Workspace не различает локальный путь и сетевой. Если хост-ОС смонтировал — Workspace работает с ним как с обычной директорией.

Импорт файлов из сетевого хранилища — тот же процесс, что и с флешки.

### 2.5 Зеркальные папки (Mirror Folders)

Папка на хост-ОС, привязанная к проекту WORKSPACE. Работает двусторонне:

- Файл добавлен в `D:\Work` на Windows → появляется в проекте WORKSPACE с тегами
- Файл получил тег в WORKSPACE → виден в папке Windows (через `.tag`-файлы или расширенные атрибуты, если хост-ОС поддерживает)
- Коллега, не использующий WORKSPACE, видит обычную папку. Ты внутри WORKSPACE видишь ту же папку, но с поиском, тегами и версионностью

**Настройка:** правый клик на проекте → «Подключить папку хост-ОС» → выбрать путь.

**Host Shim** наблюдает за путём через `inotify` / `FSEvents` / `ReadDirectoryChanges`. Storage Manager синхронизирует метаданные: новый файл на диске → создание записи в VFS, удаление → архивирование, переименование → обновление passport.

**Ограничение:** WORKSPACE не синхронизирует теги обратно в файловую систему хост-ОС, если хост не поддерживает расширенные атрибуты. В этом случае теги живут только внутри WORKSPACE (SQLite), но файлы остаются на своих местах.

## 3. Импорт файлов

### Процесс

```
1. Пользователь нажал «Импортировать файлы»
   |
   v
2. Host Shim → сканирование директории носителя
   +-- Рекурсивный обход
   +-- Определение типов файлов (MIME)
   +-- Подсчёт: количество файлов, общий размер
   |
   v
3. UI: "Найдено 12 847 файлов (28 ГБ). Импортировать?"
   +-- Фильтры: "Только фото" / "Только документы" / "Всё"
   +-- Целевой проект: текущий / другой / Home Project
   |
   v
4. Storage Manager → импорт
   +-- Копирование файла через Host Shim
   +-- Вычисление BLAKE3-хеша → проверка дубликатов
   +-- Создание записи в VFS (файл + метаданные + паспорт)
   +-- Автоматические теги по типу файла
   +-- Привязка к проекту
   |
   v
5. Search Engine → индексация (фоново)
   +-- Файлы: извлечение текста (PDF, DOCX)
   +-- Изображения: OCR (опционально)
   +-- Музыка: метаданные (ID3)
   |
   v
6. Завершение → уведомление
   "Импортировано 12 847 файлов. 340 дубликатов пропущено."
```

### Дедупликация

Перед копированием — вычисление BLAKE3-хеша файла:
- Хеш уже есть в VFS → файл не копируется, создаётся ссылка на существующий
- Пользователь видит: «340 дубликатов обнаружено, пропущено»
- Дедупликацию можно отключить в настройках импорта

### Отмена

Импорт можно отменить в любой момент:
- Уже импортированные файлы остаются в VFS
- Частично скопированный файл — удаляется
- Пользователь видит: «Импорт отменён. 4523 из 12847 файлов импортировано.»

### Через Command Bar

```
"импортировать с флешки"              → обнаружен USB → UI импорта
"импортировать фото с флешки"         → фильтр по типу: изображения
"импортировать всё с D:"              → указан конкретный путь (Windows)
"импортировать /media/usb/photos"     → указан конкретный путь (Linux)
```

Intent Parser определяет:
- Источник: носитель / путь
- Фильтр: по типу файла
- Цель: текущий проект / другой проект

## 4. Экспорт файлов

Копирование файлов из VFS на внешний носитель.

### Процесс

```
1. Пользователь: "скинуть на флешку" / контекстное меню → "Экспортировать"
   |
   v
2. Storage Manager → список подключённых носителей
   +-- Если ни одного → "Подключите USB-накопитель"
   +-- Если один → выбирается автоматически
   +-- Если несколько → выбор из списка
   |
   v
3. Копирование через Host Shim
   +-- Из VFS → во временную директорию → на носитель
   +-- Прогресс-бар: "Экспорт... 45%"
   |
   v
4. Завершение → уведомление
   "23 файла экспортировано на USB-накопитель "BACKUP-1TB""
```

### Через Command Bar

```
"скинуть на флешку"                   → экспорт выбранных файлов
"экспортировать проект на USB"        → все файлы проекта
"экспортировать все фото на флешку"   → фильтр по тегу #фото
```

### Шифрование при экспорте (опционально)

Пользователь может выбрать «Зашифровать при экспорте»:
- Файлы шифруются backup key (из recovery-фразы) перед записью
- На носителе — нечитаемые файлы без ключа
- Для восстановления: импорт → ввод recovery-фразы → расшифровка

Политика Бэка может **требовать** шифрование при экспорте (корпоративный сценарий). Owner может полностью запретить экспорт на внешние носители через `external_media_allow_export`.

Подробно: политики экспорта, шифрование, векторы атак через носители — в [Layer Security, §7.6](layer-7-security.md).

## 5. Жизненный цикл носителя

```
┌───────────┐     ┌──────────────┐     ┌────────────┐     ┌───────────┐
│ Подключён │────▶│ Обнаружен    │────▶│ Действие   │────▶│ Работа    │
│ к хост-ОС │     │ Host Shim'ом │     │ пользователя│    │ (импорт/  │
└───────────┘     └──────────────┘     └────────────┘     │ экспорт)  │
                                                            └─────┬─────┘
                                                                  │
                                                          ┌───────▼───────┐
                                                          │ Извлечён      │
                                                          │ из хост-ОС    │
                                                          └───────────────┘
```

**Состояния носителя в Storage Manager:**

| Состояние | Что |
|-----------|-----|
| Обнаружен | Host Shim определил подключение, передал информацию в Бэк |
| Готов | Пользователь получил уведомление, ожидает действие |
| Активен | Идёт импорт или экспорт |
| Простой | Действие завершено, носитель подключён, ничего не происходит |
| Извлекается | Пользователь запросил безопасное извлечение |
| Извлечён | Носитель отключён |

**Безопасное извлечение:**
- Command Bar: «извлечь флешку»
- Или: кнопка в уведомлении
- Host Shim → хост-ОС: запрос на размонтирование (Linux: `umount`, Windows: `CM_Request_Device_Eject`)
- Если идёт импорт/экспорт → предупреждение

## 6. Storage Manager — технически

### 6.1 Архитектура

```
Host Shim (Level 0, Rust):
  |
  +-- Storage Watcher:
  |   +-- Подписка на события хост-ОС (udev / WM_DEVICECHANGE / DiskArbitration)
  |   +-- Обнаружение: тип, файловая система, размер, метка
  |   +-- Передача событий в Storage Manager (Level 1)
  |
  +-- File I/O:
      +-- Чтение файлов с носителя (для импорта)
      +-- Запись файлов на носитель (для экспорта)
      +-- Через стандартные API хост-ОС

Storage Manager (Level 1, Bun):
  |
  +-- Device Registry (SQLite):
  |   +-- devices: id, type, label, path, fs_type, size, free, status, last_seen
  |   +-- rules: device_label, action (import/backup/ignore), filters
  |
  +-- Import Engine:
  |   +-- Сканирование директории → список файлов
  |   +-- Дедупликация (BLAKE3-хеш vs VFS)
  |   +-- Копирование в VFS через Host Shim
  |   +-- Создание метаданных (тип, размер, дата, теги)
  |   +-- Очередь: приоритет по размеру (мелкие первыми — быстрый прогресс)
  |
  +-- Export Engine:
  |   +-- Выбор носителя
  |   +-- Копирование из VFS через Host Shim
  |   +-- Опциональное шифрование (backup key)
  |   +-- Прогресс → UI
  |
  +-- Rule Engine:
      +-- Автоматические действия по правилам
      +-- Маппинг: device_label → action + filters
      +-- При подключении: проверка правил → автоматическое действие или уведомление

Display Server (Level 3):
  |
  +-- Уведомление: тип носителя, размер, действия
  +-- Прогресс-бар: импорт/экспорт
  +-- Workspace.Files: подключённые носители (если активны)
```

### 6.2 Импорт — потоки данных

```
USB-флешка (хост-ОС: /mnt/usb)
  |
  v
Host Shim (Level 0):
  +-- open("/mnt/usb/photo.jpg") → read() → buffer
  |
  v
Storage Manager (Level 1):
  +-- BLAKE3(buffer) → проверка дубликатов
  +-- Запись в VFS: Host Shim → write("/home/user/.WORKSPACE/vfs/abc123.jpg")
  +-- Метаданные → SQLite: file_id, name, size, hash, tags, project_id
  +-- Search Engine → индексация (фоново)
  |
  v
Фронт:
  +-- Файл виден в проекте
  +-- Файл находится через поиск
  +-- Файл доступен по тегу #фото
```

### 6.3 Где хранятся файлы VFS

VFS — это директория на диске хост-ОС, управляемая Workspace:

| Хост-ОС | Путь VFS по умолчанию |
|---------|----------------------|
| Windows | `%USERPROFILE%\.WORKSPACE\vfs\` |
| Linux | `~/.WORKSPACE/vfs/` |
| macOS | `~/.WORKSPACE/vfs/` |

Внутри — Content Addressable Store:

```
~/.WORKSPACE/vfs/
├── blobs/              ← Body: файлы по CID (BLAKE3-хеш содержимого)
│   ├── ab/             ← Префикс 2 символа для равномерного распределения
│   │   └── cdef1234... ← Blob: реальные байты файла
│   └── 12/
│       └── 3456abcd...
├── passport.sqlite     ← Passport: метаданные, теги, права, версии
├── crdt/               ← CRDT-журнал для синхронизации
└── cache/              ← Ghost-файлы и временные blob'ы
```

**Content Addressable Storage:**
- CID = BLAKE3-хеш содержимого. Два одинаковых файла = один blob на диске
- Passport (SQLite) хранит имя, проект, теги, права доступа, историю версий и CID тела
- Удаление файла из проекта ≠ удаление blob'а. Blob живёт, пока на него ссылается хотя бы один passport

**Ленивая загрузка (Lazy Load):**
- Ghost-файл: запись в passport есть, CID известен, но blob отсутствует локально
- `available_local: false` → файл виден в проекте, но помечен облачком
- При открытии: on-demand fetch через P2P Mesh с устройства, где blob есть
- Prefetch: система анализирует паттерны и подгружает файлы заранее (например, весь плейлист)

**Версионность:**
- Каждое изменение = новый blob (новый CID). Старый CID остаётся в Version Store
- Текстовые файлы: хранятся diff'ы (XOR-дельта), не полные копии — экономия места
- Версии привязаны к проекту, синхронизируются через CRDT

Пользователь **не должен** вручную работать с этой директорией. Весь доступ — через Workspace.

## 7. Безопасность

### Внешние носители — вектор атаки

| Вектор атаки | Защита |
|-------------|--------|
| Вредоносный файл на флешке | Файлы импортируются в VFS, приложения не запускают их напрямую. Приложения в V8 Isolate — изолированы. Импортированный `.exe` не выполнится в Workspace |
| Автозапуск с носителя | Нет автозапуска. Пользователь явно выбирает действие. Никакой код с носителя не выполняется автоматически |
| Огромное количество файлов (DoS) | Storage Manager лимитирует: максимальное количество файлов за один импорт (настраивается, default: 100 000), максимальный размер одного файла (default: 10 ГБ) |
| Подмена носителя (BadUSB) | Host Shim определяет тип устройства. Если «флешка» — клавиатура — предупреждение: «Устройство определяет себя как клавиатуру. Возможна подмена.» |
| Кража данных через экспорт | Политика Бэка: Owner может запретить экспорт на внешние носители. Корпоративный Бэк: `allow_external_export: false` |

### Политики Бэка (настраивает Owner)

| Настройка | Что | По умолчанию |
|-----------|-----|-------------|
| `allow_import` | Разрешить импорт с внешних носителей | `true` |
| `allow_export` | Разрешить экспорт на внешние носители | `true` |
| `require_encrypt_export` | Требовать шифрование при экспорте | `false` |
| `max_import_files` | Максимум файлов за один импорт | `100000` |
| `max_import_file_size` | Максимум размер одного файла | `10GB` |

### Пользовательские настройки

| Настройка | Что | По умолчанию |
|-----------|-----|-------------|
| Дедупликация при импорте | Пропускать дубликаты | Вкл |
| Автоматические теги | Ставить теги по типу файла | Вкл |
| Индексация после импорта | Индексировать содержимое файлов | Вкл |
| Уведомление о носителе | Показывать уведомление при подключении | Вкл |

## 9. Сетевые подключения WORKSPACE-to-WORKSPACE: P2P Mesh

Связь между устройствами с Workspace. Реализована полностью внутри Workspace, не зависит от хост-ОС (кроме IP-соединения).

Подробно: [Layer: Подсистемы, секция 9](layer-8-technical-decomposition.md).

**Сводка:**

| Возможность | Технология | Уровень |
|------------|-----------|---------|
| Обнаружение устройств в LAN | mDNS (Multicast DNS) | Level 2 |
| Обнаружение устройств в интернете | libp2p DHT + Gossip | Level 2 |
| Зашифрованные туннели | WireGuard | Level 0 |
| Синхронизация данных | CRDT (Causal Trees, LWW) | Level 2 |
| Стриминг медиа | P2P (WireGuard tunnel) | Level 2 |
| NAT traversal | Tailscale-подобный механизм | Level 2 |
| Авторизация нового устройства | QR-код + обмен ключами + ZKP | Level 1 + Level 2 |

**Что делает Mesh Engine:**
- Обнаруживает другие WORKSPACE-устройства в локальной сети (mDNS)
- Устанавливает WireGuard-туннели между устройствами
- Синхронизирует данные через CRDT
- Стримит медиа между устройствами (фильм с ПК → телефон)
- Работает без интернета (прямая передача по Wi-Fi / LAN)

**Что Mesh Engine НЕ делает:**
- Не работает с не-WORKSPACE устройствами (SMB-шары, принтеры, Smart TV без WORKSPACE)
- Не реализует DLNA, AirPlay, Chromecast, SMB, NFS, IPP

## 10. Сетевые подключения WORKSPACE-to-World: через хост-ОС

### 10.1 SMB/NFS — файловые шары

Другой ПК в сети расшарил папку по SMB (Windows) или NFS (Linux). Workspace хочет получить доступ к файлам.

**Как это работает:**

```
Другой ПК: \\PC-OFFICE\Documents (SMB-шара)
  |
  v
Хост-ОС:
  +-- Монтирует SMB-шару как файловую систему
  |   +-- Windows: net use Z: \\PC-OFFICE\Documents
  |   +-- Linux: mount -t cifs //pc-office/Documents /mnt/docs
  |   +-- macOS: mount_smbfs //pc-office/Documents /Volumes/docs
  |
  v
Host Shim (Level 0):
  +-- Видит смонтированный путь как обычную директорию
  +-- Передаёт в Storage Manager как «сетевой носитель»
  |
  v
Workspace:
  +-- Пользователь видит: «Обнаружена сетевая папка "Documents"»
  +-- Действия: Импортировать файлы / Игнорировать
  +-- Импорт — тот же механизм что с USB-флешкой
```

**Подключение сетевой папки — через Command Bar:**

```
"подключить сетевую папку \\PC-OFFICE\Documents"
"подключить //nas.local/music"
"подключить smb://192.168.1.50/shared"
```

Intent Parser → Host Shim → хост-ОС монтирует → Storage Manager → уведомление → импорт.

**Отключение:**

```
"отключить сетевую папку Documents"
```

Host Shim → хост-ОС размонтирует.

**Ограничения:**
- Workspace **не реализует** SMB/NFS протокол — полагается на хост-ОС
- Если хост-ОС не поддерживает SMB (маловероятно) — сетевая папка недоступна
- Аутентификация (логин/пароль для SMB) — через Host Shim → диалог хост-ОС
- Сетевая папка — **не постоянное хранилище** VFS. Только импорт. Файлы копируются в VFS, после чего сетевая папка может быть отключена

**Автоматическое обнаружение:**

Host Shim может опрашивать хост-ОС о доступных сетевых ресурсах:
- Windows: `NetShareEnum` — список SMB-шар в сети
- Linux: `avahi-browse` — обнаружение сервисов через mDNS/DNS-SD
- macOS: `NSNetServiceBrowser` — Bonjour-обнаружение

При обнаружении новой шары — уведомление пользователю.

### 10.2 DLNA — за рамками

DLNA/UPnP **не поддерживается** Workspace. Причины:

- Протокол DLNA сложен, плохо стандартизирован, реализация тянет зависимостей
- Основной сценарий «медиа на телевизоре» покрывается WORKSPACE TV — собственное приложение, P2P-стриминг с WORKSPACE Base
- Медиасерверы в сети доступны через SMB/NFS — файлы можно импортировать

**Альтернативы:**
- Smart TV → WORKSPACE TV приложение → P2P-стриминг с WORKSPACE Base
- Медиасервер (NAS) → SMB/NFS шара → импорт файлов в VFS → воспроизведение через WORKSPACE Player
- Если хост-ОС имеет DLNA-клиент — Workspace не участвует, пользователь работает через хост-ОС напрямую

### 10.3 Сетевые принтеры — через хост-ОС

Workspace **не реализует** протоколы принтеров (IPP, LPD, JetDirect). Хост-ОС управляет драйверами, очередью печати, обнаружением принтеров.

**Как это работает:**

```
Сетевой принтер (192.168.1.100, IPP)
  |
  v
Хост-ОС:
  +-- Обнаруживает принтер (mDNS / SNMP)
  +-- Устанавливает драйвер
  +-- Управляет очередью печати
  |
  v
Host Shim (Level 0):
  +-- Вызов API печати хост-ОС:
  |   +-- Windows: WritePrinter / StartDocPrinter
  |   +-- Linux: CUPS API (libcups)
  |   +-- macOS: CUPS API
  |
  v
Workspace:
  +-- Command Bar: "напечатать" / "напечатать документ X"
  +-- Intent Parser → Host Shim → хост-ОС → принтер
  +-- Workspace не знает какой принтер, какой драйвер, какой протокол
  +-- Workspace знает только: «отправить документ на печать»
```

**Command Bar:**

```
"напечатать"                     → печать текущего документа
"напечатать смету.pdf"           → печать конкретного файла
"напечатать фото со стройки"     → поиск + печать
```

**Ограничения:**
- Workspace не настраивает принтеры — это задача хост-ОС
- Если принтер не настроен в хост-ОС — Workspace не сможет печатать
- Workspace не показывает список принтеров, не управляет очередью — это делает хост-ОС
- Диалог «выбор принтера» — через Host Shim → системный диалог хост-ОС

### 10.4 Другие сетевые сервисы

**AirPlay / Chromecast:**
- Не реализуются Workspace
- Если хост-ОС поддерживает (macOS — AirPlay нативно, Windows — сторонние приложения) — Workspace не участвует
- Пользователь работает через хост-ОС напрямую

**Скanners:**
- Аналогично принтерам — через хост-ОС
- Host Shim → TWAIN/SANE API хост-ОС → сканер
- Workspace: «отсканировать» → Host Shim → хост-ОС → файл → импорт в VFS

**VPN:**
- Workspace использует свой WireGuard для P2P Mesh — это внутренний VPN Workspace
- Корпоративный VPN (OpenVPN, IPSec, WireGuard) — настраивается на уровне хост-ОС
- Workspace видит результат: «есть связь с корпоративным Бэком / нет связи»

## 11. Сетевая конфигурация

### Wi-Fi / Ethernet

Workspace **не управляет** сетевыми интерфейсами. Хост-ОС подключается к Wi-Fi, настраивает Ethernet, управляет маршрутизацией.

**Что видит Workspace:**

```
Host Shim → хост-ОС API:
  +-- Список сетевых интерфейсов (есть / нет)
  +-- Статус подключения (подключён / отключён / ограниченный)
  +-- Тип подключения (Wi-Fi / Ethernet / Cellular)
  +-- IP-адрес (для P2P Mesh)
  +-- Есть ли интернет (проверка: ping relay.WORKSPACE.app)
```

**Workspace знает только:**
- Есть подключение к локальной сети → mDNS-обнаружение работает, P2P Mesh работает
- Есть интернет → удалённые Бэки доступны, relay доступен, обновления скачиваются
- Нет сети → оффлайн-режим, работа с кэшем

**Настройка Wi-Fi — через хост-ОС:**
- Workspace не показывает список Wi-Fi сетей
- Подключение к Wi-Fi — задача хост-ОС (Windows: системные настройки, Linux: NetworkManager, macOS: системные настройки)
- Если Wi-Fi не подключён — Workspace показывает: «Нет подключения к сети. Подключитесь к Wi-Fi в настройках устройства.»

### Прокси

- Прокси настраивается на уровне хост-ОС
- Workspace подхватывает настройки прокси через Host Shim:
  - Windows: `InternetQueryOption` → `INTERNET_OPTION_PROXY`
  - Linux: `http_proxy` / `https_proxy` environment variables
  - macOS: `SystemConfiguration` framework → proxy settings
- P2P Mesh (WireGuard) игнорирует системный прокси — работает напрямую

### Диагностика

Command Bar:

```
"статус сети"                     → Host Shim → проверка подключения → результат в UI
  "Локальная сеть: подключена (Wi-Fi, 192.168.1.15)"
  "Интернет: доступен"
  "P2P Mesh: 3 устройства (ПК, телефон, WORKSPACE Base)"
  "WORKSPACE Base: подключён (192.168.1.100)"

"проверить связь с WORKSPACE Base"     → ping через WireGuard → результат
"почему нет синхронизации"        → диагностика Mesh Engine → причина в UI
```

## 12. Host Shim — сетевой уровень

### Что Host Shim делает

| Функция | API хост-ОС | Что Workspace получает |
|---------|-----------|---------------------|
| Монтирование SMB/NFS | `mount` / `net use` | Путь к файлам → Storage Manager |
| Размонтирование SMB/NFS | `umount` / `net use /delete` | Событие «носитель отключён» |
| Обнаружение сетевых шар | `NetShareEnum` / `avahi-browse` | Список доступных шар → уведомление |
| Печать | `WritePrinter` / CUPS | Отправка документа на принтер |
| Сканирование | TWAIN / SANE | Файл со сканера → импорт |
| Статус сети | `GetAdaptersInfo` / `getifaddrs` | Подключён / отключён, IP, тип |
| Проверка интернета | HTTP-запрос к relay.WORKSPACE.app | Доступен / недоступен |
| Настройки прокси | `InternetQueryOption` / env vars | Адрес прокси |

### Что Host Shim НЕ делает

- Не реализует SMB/NFS/DLNA/IPP/AirPlay/Chromecast протоколы
- Не управляет Wi-Fi/Ethernet подключениями
- Не настраивает VPN (кроме WireGuard для P2P Mesh)
- Не управляет драйверами принтеров/сканеров
- Не показывает системные диалоги (выбор принтера, ввод пароля SMB) — вызывает диалоги хост-ОС
